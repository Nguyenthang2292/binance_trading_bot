# Trend Breakout Multi-Timeframe v1.1 Design

**Date:** 2026-05-16  
**Status:** Proposed, not implemented  
**Applies to:** `trend_breakout` strategy plugin, SignalEngine order flow, runtime logging  
**Related docs:**

- `docs/strategies/trend_breakout.md`
- `docs/design/strategies/2026-05-15-strategy-trend-breakout-v1.0.md`
- `docs/archives/design/2026-05-14-market-scanner-strategy-engine-v1.0.md`

---

## Understanding Summary

- The current `trend_breakout` plugin only evaluates `4h`; it ignores every other interval even if configured.
- The requested change is to add signal handling for `30m` and `1h`, while preserving `4h`.
- Each order must be traceable on the command line to the strategy name and the timeframe that produced it.
- Trailing stop behavior must follow the signal timeframe: `30m` signal uses `30m` trailing, `1h` uses `1h`, `4h` uses `4h`.
- This is an extension of the existing strategy, not a new separate strategy type.
- The current one-position-per-symbol tracker behavior remains in force unless redesigned later.
- No implementation is included in this design task.

## Assumptions

- The same Donchian breakout rule applies to all enabled timeframes.
- Default enabled intervals for `trend_breakout` become `["30m", "1h", "4h"]`.
- `breakout_period`, `atr_period`, `risk_pct`, `sl_multiplier`, `takeProfitPercent`, `tp_multiplier`, and `min_confidence` remain shared across timeframes for v1.1.
- Scanner must warm up and stream `30m`, `1h`, and `4h` klines.
- Order attribution can be carried through `OrderMetadata.strategyTag` and log strings without changing Binance order APIs.
- The tracker stores one position per symbol, so a symbol cannot hold separate `30m` and `4h` positions at the same time.

## Non-Functional Requirements

- **Performance:** Adding `1h` increases scanner subscriptions and warmup requests. The current scanner concurrency and stream splitting should handle one additional interval, but logs should make warmup size visible.
- **Reliability:** Missing kline data for one timeframe must not break evaluation for other timeframes.
- **Safety:** No silent order attribution. Every open order generated by SignalEngine must include strategy and timeframe in logs and metadata.
- **Maintainability:** Avoid duplicating strategy classes per timeframe. Use config-driven interval handling.
- **Operational clarity:** CMD output must let the operator answer: which strategy, which timeframe, which symbol, which direction, and why.

## Decision Log

| Decision | Alternatives | Rationale |
|---|---|---|
| Extend existing `trend_breakout` plugin to multi-timeframe | Create separate strategy types per TF | Existing strategy logic is identical across TFs, so config-driven intervals are simpler. |
| Use option B: trailing timeframe equals signal timeframe | Keep trailing fixed at `4h`; per-TF custom configs | Matches operator request and keeps behavior intuitive. |
| Keep one-position-per-symbol behavior | Allow one position per symbol per TF | Current tracker and risk controls are symbol-scoped. Per-TF positions require a larger position identity redesign. |
| Add strategy and TF to order metadata/logs | Encode only in clientOrderId | Metadata/logs are cleaner and avoid clientOrderId length/format risk. ClientOrderId can remain stable. |
| Shared parameters across TFs for v1.1 | Per-TF breakout/risk/trailing config | Shared params are lower risk. Per-TF tuning should be a later v1.2 after live/backtest evidence. |
| Evaluate before tracker skip when skip logging is required | Keep current pre-evaluation `tracker.has(symbol)` fast path | To log `position already tracked` only for real signals, the engine must evaluate the signal first, then check tracker before order placement. |

## Design Approaches Considered

### Approach A: Fixed 4h Trailing

Entry signals run on `30m`, `1h`, and `4h`, but trailing stop always uses `4h`.

Pros:

- Minimal engine changes.
- Stable slower trailing behavior.

Cons:

- A `30m` entry can wait too long for trailing updates.
- Does not match the requested behavior.

### Approach B: Signal Timeframe Drives Trailing

Entry signal runs on `30m`, `1h`, and `4h`. The position records the interval that created it. Trailing stop uses that same interval.

Pros:

- Matches operator intent.
- Logs and behavior are easy to reason about.
- Does not require separate strategy instances.

Cons:

- More frequent trailing changes for `30m` positions.
- Requires SignalEngine to pass interval context into position creation.

### Approach C: Per-Timeframe Strategy Config

Each timeframe has separate breakout period, risk, TP/SL, trailing candles, and scan cadence.

Pros:

- Most flexible.
- Better for tuned production strategies later.

Cons:

- Higher config complexity.
- More testing and operational burden.
- Premature for the current request.

**Selected approach:** Approach B.

## Current State

### Plugin

`plugins/src/trend_breakout/strategy_trend_breakout.cpp` currently contains:

```cpp
constexpr std::string_view kTradingInterval = "4h";
```

and returns no signal when:

```cpp
if (interval != kTradingInterval) {
    return {};
}
```

This hard gate must be removed or replaced by config-driven interval validation.

### Runtime Config

Current strategy config has:

```json
"intervals": ["4h"]
```

Scanner config currently has:

```json
"intervals": ["15m", "30m", "4h"]
```

For v1.1, scanner should include `1h`, and the strategy should include `30m`, `1h`, and `4h`.

### Engine

`WorkQueue` sends `(symbol, interval, strategy)` to SignalEngine. SignalEngine currently receives the interval in `processItem`, but `openPosition` only receives symbol, direction, ATR, price, and strategy config. The order placement layer therefore loses the exact signal interval.

## Warmup and Candle Coverage

The strategy needs enough candles for both:

- Donchian breakout lookback: `breakout_period`
- ATR calculation: `atr_period + 1` minimum for a non-zero ATR seed

The current plugin also evaluates the most recent closed candle while allowing `klines.back()` to be a still-forming candle, so the existing implementation requires:

```text
min_candles = breakout_period + 2
```

With defaults:

```text
breakout_period = 20
atr_period = 14
min_candles = max(20 + 2, 14 + 1) = 22
```

Time coverage for the 20-candle breakout window is approximately:

| Timeframe | 20-candle breakout window | 22-candle implementation minimum |
|---|---:|---:|
| `30m` | 10 hours | 11 hours |
| `1h` | 20 hours | 22 hours |
| `4h` | 80 hours | 88 hours |

Current scanner config uses:

```json
"warmup_initial_limit": 99,
"kline_buffer_size": 200
```

This is sufficient for `30m`, `1h`, and `4h` because warmup fetches 99 candles per configured interval. Approximate warmup coverage is:

| Timeframe | 99-candle warmup coverage |
|---|---:|
| `30m` | 49.5 hours |
| `1h` | 99 hours |
| `4h` | 396 hours |

Implementation must preserve this coverage when adding `1h`. Startup logs already print `initial_limit`; implementation should additionally validate that `warmup_initial_limit >= min_candles` for every enabled strategy interval, or warn loudly if it is not.

## Plugin Design

### Interval Handling

The plugin should trust `StrategyConfig.intervals` as the allowed set. Evaluation should return no signal when the current interval is not configured.

Pseudo behavior:

```cpp
if (!contains(m_cfg.intervals, interval)) {
    return {};
}
```

The old hard-coded `4h` gate should be removed.

### Signal Reason

Signal reason must include the interval:

```text
30m Donchian breakout long: close=... > high20=...
1h Donchian breakout short: close=... < low20=...
4h Donchian breakout long: close=... > high20=...
```

This gives direct CMD visibility when SignalEngine logs the reason.

### Default Intervals

If config omits intervals, v1.1 default should be:

```cpp
{"30m", "1h", "4h"}
```

This prevents a misconfigured empty strategy from silently remaining `4h` only.

## SignalEngine Design

### Carry Signal Context

SignalEngine should preserve the interval from `WorkItem` through order creation.

Current shape:

```cpp
openPosition(symbol, direction, atr, currentPrice, cfg)
```

Proposed shape:

```cpp
openPosition(symbol, interval, direction, atr, currentPrice, cfg, signal.reason)
```

The engine does not need to own strategy-specific rules. It only carries context.

### Evaluation Order

Current code checks:

```cpp
if (m_tracker.has(item.symbol)) {
    co_return;
}
```

before strategy evaluation. That is efficient, but it cannot distinguish "no signal" from "valid signal skipped because the symbol already has a tracked position".

For v1.1 logging, the order should become:

1. Load klines for `(symbol, interval)`.
2. Evaluate strategy.
3. If no signal, return silently.
4. If signal confidence/ATR fails, return silently or with debug-only diagnostics.
5. Check `m_tracker.has(symbol)`.
6. If tracked, log `signal skipped ... reason="position already tracked"` and return.
7. If not tracked, continue to sizing, exposure, and order placement.

This must be implemented carefully so `openPosition` is never called before the tracker check.

### Tracked Position

`TrackedPosition` should store:

```cpp
std::string strategyName;
std::string signalInterval;
std::string signalReason;
```

When opening a position:

```cpp
tracked.strategyName = cfg.name;
tracked.signalInterval = interval;
tracked.signalReason = signal.reason;
tracked.trailingInterval = interval;
```

If `cfg.trailingStop.enabled` is false, trailing remains disabled.

### Trailing Interval Rule

For option B:

```cpp
tracked.trailingInterval = signalInterval;
```

Fallback:

```cpp
if signalInterval == "":
    use cfg.trailingStop.interval
else:
    use signalInterval
```

`signalInterval` is a `std::string`. "Empty" means the string value is exactly `""`, not an uninitialized object. It can occur only through legacy paths, reconstructed positions, or tests that do not yet populate the new field. This protects existing tests and legacy behavior.

## Logging Design

### Signal Log

When a signal passes confidence and ATR checks, log one line before sizing:

```text
strategy signal strategy="Trend Breakout Trader" tf=30m symbol=JTOUSDT direction=Long confidence=1.00 atr=0.123 reason="30m Donchian breakout long: ..."
```

Fields:

- `strategy`
- `tf`
- `symbol`
- `direction`
- `confidence`
- `atr`
- `reason`

### Market Order Log

Before placing market order:

```text
opening position strategy="Trend Breakout Trader" tf=30m symbol=JTOUSDT side=BUY qty=... notional=...
```

### Existing Order Placement Log

Current order placement logs are emitted by order services and do not include strategy metadata. Add metadata to log output when available:

```text
order_placement ... symbol=JTOUSDT ... state=Accepted ... strategy="Trend Breakout Trader" tf=30m
```

If only one string field is feasible in v1.1, use a compact tag:

```text
strategyTag=Trend Breakout Trader|tf=30m
```

Preferred design is explicit fields in the log line.

### Skip Logs

Do not log every no-signal symbol by default. That would be too noisy for 531 symbols x 3 intervals.

Log only:

- accepted signal
- blocked by exposure
- scaled too small
- order rejected
- position skipped because symbol already tracked, if a valid signal existed

## Order Metadata Design

Use existing `OrderMetadata`:

```cpp
struct OrderMetadata {
    std::optional<int64_t> magic;
    std::optional<std::string> comment;
    std::optional<std::string> strategyTag;
};
```

Proposed values:

```cpp
metadata.strategyTag = "Trend Breakout Trader";
metadata.comment = "tf=30m reason=Donchian breakout";
```

For market order, TP order, stop-loss order, and trailing-stop replacement, metadata should carry the same strategy/TF context.

If later a structured metadata type is needed, add fields:

```cpp
std::optional<std::string> strategyName;
std::optional<std::string> signalInterval;
```

For v1.1, reuse existing fields to minimize blast radius.

## Data Flow

1. Scanner warms up symbols for `30m`, `1h`, `4h`.
2. WorkQueue builds items for each symbol, interval, and active strategy.
3. SignalEngine calls `strategy.evaluate(symbol, interval, klines)`.
4. Plugin checks whether interval is configured.
5. Plugin evaluates Donchian breakout on that interval.
6. SignalEngine logs accepted signal with strategy and TF.
7. SignalEngine sizes order and checks exposure.
8. SignalEngine opens market order with metadata.
9. SignalEngine places TP with same metadata.
10. SignalEngine tracks position with `strategyName` and `signalInterval`.
11. TrailingStopController evaluates using `tracked.trailingInterval`, which equals the signal TF.

## Edge Cases

### Same Symbol Signals on Multiple TFs

Current tracker is symbol-scoped. If `JTOUSDT` has a `30m` position, later `1h` or `4h` signals for `JTOUSDT` are skipped.

Expected behavior for v1.1:

```text
signal skipped strategy="Trend Breakout Trader" tf=1h symbol=JTOUSDT reason="position already tracked"
```

Only log this when there is an actual signal, not before evaluation.

### Missing 1h Scanner Data

If scanner config does not include `1h`, WorkQueue should not create `1h` items. Strategy config alone is not enough.

Operational config must keep scanner and strategy intervals aligned.

Startup validation should warn when a strategy interval is not present in scanner intervals:

```text
strategy interval not scanned strategy="Trend Breakout Trader" interval=1h
```

For v1.1 this should be a warning, not a hard failure, because existing configurations may be temporarily mismatched during rollout.

### Trailing Disabled

If `trailing_enabled=false`, no trailing stop should be maintained even if signal interval exists.

### Fixed `trailing_interval` in Old Config

Existing config may still contain:

```json
"trailing_interval": "4h"
```

For v1.1 option B, implementation should prefer signal interval. The fixed config value can be left as fallback for compatibility.

### Position Tracker Thread Safety

`PositionTracker` currently uses a mutex around add/remove/read operations. Multi-timeframe dispatch increases the number of paths that can reach the tracker, but it does not by itself require a new concurrency model.

Implementation should still verify:

- `has(symbol)` and `add(position)` cannot race into duplicate opens for the same symbol.
- User-data fill handling, time-exit, and trailing-stop updates remain safe while scan processing is active.
- If evaluation is moved before `tracker.has(symbol)`, the final tracker check still happens immediately before order placement.

If future work allows one position per symbol per timeframe, this mutex-protected symbol map must be redesigned around a richer position key.

### Metadata Comment Usage

Before relying on `OrderMetadata.comment` for `tf=...`, implementation should grep every order path and journal path that preserves metadata:

- market orders
- normal limit orders
- stop/protection algo orders
- trailing stop replacement
- MQL4 adapter paths
- order journal serialization/deserialization

If any path drops `comment`, use `strategyTag` as the minimum reliable tag and extend logging directly in the order placement result path.

## Testing Strategy

### Plugin Unit Tests

Add or update tests proving:

- `30m` breakout can emit Long and Short.
- `1h` breakout can emit Long and Short.
- `4h` breakout still works.
- unconfigured interval returns no signal.
- signal reason includes timeframe.

### SignalEngine Tests

Add tests proving:

- `openPosition` stores `strategyName` and `signalInterval`.
- tracked trailing interval equals signal interval.
- market order metadata includes strategy and TF.
- TP metadata includes strategy and TF.
- log helper formats strategy and TF consistently, if log helper is extracted.

### Config Test or Smoke Test

Validate that runtime config includes:

```json
"scanner": { "intervals": ["30m", "1h", "4h"] }
"strategies[0].intervals": ["30m", "1h", "4h"]
```

### Regression Tests

Existing tests for:

- tick-size price formatting
- step-size quantity formatting
- min-notional sizing
- exposure blocking and scale-down

must continue to pass.

## Implementation Plan

This section is a handoff plan only. It is not implementation.

1. Update strategy config defaults and remove hard-coded `4h` interval gate.
2. Update `config.json` scanner intervals to include `1h` and strategy intervals to `["30m", "1h", "4h"]`.
3. Pass `WorkItem.interval` into `openPosition`.
4. Add strategy and interval fields to tracked position state.
5. Set `tracked.trailingInterval = signal interval`.
6. Attach metadata to market, TP, SL, and trailing replacement orders.
7. Extend order placement log output to include metadata strategy and TF.
8. Add signal acceptance log with strategy, TF, symbol, direction, ATR, confidence, and reason.
9. Add tests described above.
10. Rebuild plugin DLL and main executable.
11. Add startup validation that warns when `strategy.intervals` is not a subset of `scanner.intervals`.
12. Confirm tracker thread-safety under multi-TF dispatch and keep the final `has(symbol)` check immediately before order placement.
13. Grep and confirm `OrderMetadata.comment` is preserved by every order path; if not, log TF/strategy directly from SignalEngine and use `strategyTag` as the reliable metadata field.

## Acceptance Criteria

- Runtime strategy status shows intervals `[30m, 1h, 4h]`.
- Logs include accepted signal lines with `strategy` and `tf`.
- Order placement logs include strategy name and timeframe.
- A `30m` signal creates a tracked position whose trailing interval is `30m`.
- A `1h` signal creates a tracked position whose trailing interval is `1h`.
- A `4h` signal keeps existing behavior with trailing interval `4h`.
- Existing safety tests and order formatting tests pass.

---

## Risks

- More signals can increase order frequency and exposure pressure.
- `30m` trailing may move stops more often and create more cancel/replace activity.
- One-position-per-symbol behavior may hide higher timeframe signals while a lower timeframe position is open.
- Shared parameters may not be optimal for all TFs; later backtesting may justify per-TF config.

## Deferred Work

- Per-timeframe risk and breakout parameters.
- Per-timeframe position identity.
- Strategy priority when multiple TFs signal on the same symbol.
- Backtest report comparing `30m`, `1h`, and `4h` variants.

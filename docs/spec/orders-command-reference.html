Actual API in src/orders — based on current implementation.

Scope

NormalOrderService

Market / limit / close-by-market, amend, cancel, query, open orders, history, and batch operations (up to 5).

AlgoOrderService

Stop-entry / protection (SL/TP style), cancel/query by algo identity, separate journal & reconcile semantics.

  • orders::mql4::Mql4Adapter maps MQL4 operations to the typed API above.

Algo Commands

Command Method Lifecycle
Stop entry Orders::stopEntry(StopEntryDraft) Algo placement + journal/reconcile
Protection (SL/TP style) Orders::protection(ProtectionOrderDraft) Algo placement + journal/reconcile
Cancel by algoId Orders::cancelAlgoByAlgoId(symbol, algoId) Algo cancel

Query by algoId Orders::queryAlgoByAlgoId(symbol, algoId) Algo query
Query by clientAlgoId Orders::queryAlgoByClientAlgoId(symbol, clientAlgoId) Algo query
  • Placement states: Accepted · Rejected · UnknownPendingReconcile
  • When journal is configured, algo intent is recorded with orderCategory="algo" and metadata.
  • After stopEntry / protection the result does not expose a separate algoId. Use clientAlgoId with queryAlgoByClientAlgoId(…) to retrieve exchange identity.

Strict Validator Contract

Rules

  • reduceOnly=true is rejected in Hedge mode (market + limit).
  • Raw key regex:
    ^[A-Za-z][A-Za-z0-9_]{0,63}$

Blocked Raw Keys

symbol side type quantity price positionSide newClientOrderId newOrderRespType timeInForce reduceOnly clientAlgoId
recvWindow / timestamp / signature are also blocked when allowRawTimestampOverride=false.

Advisory Issues (always attached)

warning: position_mode_unknown warning: no_client_id_namespace skipped: exchange_info_unavailable

NormalPlacementResult::validation is always present on both success and failure paths.

MQL4 Adapter Mapping

orders::mql4::Mql4Adapter::orderSend(MappedOrderSendDraft)

Buy / Sell → Orders::market
BuyLimit / SellLimit → Orders::limit
BuyStop / SellStop → Orders::stopEntry
  • metadata is not dropped — passed into the typed draft and the journal.
  • stopLoss / takeProfit are not dropped. After an entry is Accepted, the adapter creates separate protection orders via Orders::protection:
    • stopLoss → ProtectionKind::StopLoss (STOP_MARKET)
    • takeProfit → ProtectionKind::TakeProfit (TAKE_PROFIT_MARKET)
    If attaching protection fails, the entry result is still returned and a warning is added to validation.issues.
  • BuyStop / SellStop optional limitPrice:
    • No limitPrice → stop-market (STOP_MARKET)
    • With limitPrice → stop-limit (STOP + price)

Journal Notes

  • The normal flow enforces the durable-journal requirement when allowBestEffortJournal=false.
  • The algo flow supports full journal/reconcile behavior when a journal is configured. Without a journal, placement is still allowed to preserve backward compatibility.

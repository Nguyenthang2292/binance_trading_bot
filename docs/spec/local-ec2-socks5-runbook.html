Run Local Bot Through EC2 Static IP

Use this runbook when the bot runs on a local Windows machine but every Binance REST and WebSocket connection must egress through an EC2 Elastic IP that is whitelisted in Binance.

Purpose

The Binance whitelist must contain the EC2 Elastic IP, not the local ISP IP. A local run is valid only when the bot logs Using SOCKS5 proxy... before connecting to Binance.

The bot supports SOCKS5 at the transport layer through BINANCE_SOCKS5_PROXY. It does not rely on operating-system transparent routing.

Traffic Model

Local Bot Runs binance_trading_bot.exe and signs requests locally.
SSH SOCKS5 Tunnel Listens on 127.0.0.1:1080 and forwards via EC2.
Binance Sees the EC2 Elastic IP as the source IP. 
binance_trading_bot.exe
  -> SOCKS5 127.0.0.1:1080
  -> SSH tunnel
  -> EC2 Elastic IP
  -> Binance Futures REST / WebSocket

Prerequisites

  • EC2 instance is running and has an Elastic IP.
  • Binance API key whitelist includes the EC2 Elastic IP.
  • Local machine has SSH access to EC2 with the repo-local private key at .\infra\binance-bot-key.pem.
  • For EC2 Instance Connect reconnects, .env contains AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, optional AWS_SESSION_TOKEN, and AWS_REGION.
  • BINANCE_API_KEY and BINANCE_SECRET_KEY are set in the bot terminal.
  • For live trading, BINANCE_TESTNET is not set. For testnet, set BINANCE_TESTNET=1.

PowerShell Run

Portable repo-relative path

Terminal 1: Open Tunnel

cd "<repo-root>"

ssh -i ".\infra\binance-bot-key.pem" `
  -D 1080 `
  -N `
  -o ExitOnForwardFailure=yes `
  -o ServerAliveInterval=30 `
  -o ServerAliveCountMax=3 `
  ubuntu@47.131.185.59

Terminal 2: Run Bot

cd "<repo-root>"

$env:BINANCE_API_KEY="<your_api_key>"
$env:BINANCE_SECRET_KEY="<your_secret_key>"
$env:BINANCE_SOCKS5_PROXY="socks5://127.0.0.1:1080"
$env:BINANCE_REQUIRE_SOCKS5_PROXY="1"

$bot = ".\build\windows-msvc-debug\bin\Debug\binance_trading_bot.exe"
& $bot

BINANCE_REQUIRE_SOCKS5_PROXY=1 is a safety guard. If the tunnel env var is missing, the bot exits before sending Binance traffic from the local IP.

Reconnect After Reboot

Use this path after Windows restarts or the SSH process exits. It reads AWS credentials from .env, detects the current local public IP, adds a narrow SSH ingress rule for that IP, creates a temporary EC2 Instance Connect key, and opens the SOCKS5 tunnel on 127.0.0.1:1080.

One-command reconnect

On TVKT-TAC

cd "D:\NGUYEN QUANG THANG\binance_trading_bot"

powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\connect_ec2_socks.ps1

On this local machine

cd "C:\Users\Admin\Desktop\binance_trading_bot"

powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\connect_ec2_socks.ps1

Expected successful output:

SOCKS tunnel is listening on 127.0.0.1:1080
SSH process id: <pid>
Proxy egress IP: 47.131.185.59

If the tunnel is already running, the script exits without creating a duplicate:

SOCKS port 1080 is already listening. Owning process: <pid>

What the script fixes automatically

  • Local ISP public IP changed after router or Windows restart.
  • EC2 security group is missing the current /32 source IP for SSH.
  • The original private key is unavailable locally, by using EC2 Instance Connect with a temporary key.
  • Port 1080 needs a fresh hidden SSH process after reboot.

Stop the tunnel

Get-NetTCPConnection -LocalPort 1080 -State Listen |
  Select-Object LocalAddress,LocalPort,OwningProcess

Stop-Process -Id <OwningProcess>

Verification

Verify the tunnel egress IP

curl.exe --socks5-hostname 127.0.0.1:1080 https://api.ipify.org

The output must equal the EC2 Elastic IP that is whitelisted in Binance: 47.131.185.59.

Verify bot environment

$env:BINANCE_SOCKS5_PROXY
$env:BINANCE_REQUIRE_SOCKS5_PROXY

Expected values:

socks5://127.0.0.1:1080
1

Verify the bot did not bypass EC2

Check trading_bot.log after startup:

Select-String -LiteralPath trading_bot.log -Pattern "Using SOCKS5 proxy" | Select-Object -Last 1

Expected Logs

Log Pattern Meaning
Using SOCKS5 proxy from BINANCE_SOCKS5_PROXY: 127.0.0.1:1080 Correct. The bot is configured to use the EC2 tunnel.
Loaded Windows root certificates into OpenSSL store Correct on Windows. OpenSSL can verify Binance TLS certificates with the Windows root store.
SOCKS5 proxy is not configured Wrong for EC2 whitelist runs. The bot will use the local egress IP.
BINANCE_REQUIRE_SOCKS5_PROXY is set but no SOCKS5 proxy was configured Guard worked. Set BINANCE_SOCKS5_PROXY and reopen the tunnel.
binanceCode=-2015 Usually invalid key, wrong IP whitelist, missing futures permission, or wrong testnet/live key.

Troubleshooting

Symptom Check Fix
Binance returns -2015 Run curl.exe --socks5-hostname 127.0.0.1:1080 https://api.ipify.org. Whitelist the returned EC2 Elastic IP and enable Futures permissions on the key.
No Using SOCKS5 proxy log Inspect $env:BINANCE_SOCKS5_PROXY in the same terminal that starts the bot. Set BINANCE_SOCKS5_PROXY=socks5://127.0.0.1:1080.
Bot exits before ping Look for BINANCE_REQUIRE_SOCKS5_PROXY guard error. Open the SSH tunnel and set BINANCE_SOCKS5_PROXY.
SOCKS5 connect fails Check Terminal 1 is still running and SSH did not exit. Restart SSH with -o ExitOnForwardFailure=yes.
SSH timeout after reboot Run Invoke-RestMethod https://checkip.amazonaws.com and compare it with the EC2 security group SSH source rules. Run .\tools\connect_ec2_socks.ps1 to add the current /32 source IP and recreate the tunnel.
Testnet key fails on live endpoint Check $env:BINANCE_TESTNET. Use matching key type and endpoint mode.

Project Scope

The main executable path routes these through ContextConfig.socks5Proxy when configured:

  • REST market data, account, orders, and listen key calls.
  • Market scanner warmup and backfill REST calls.
  • Market data WebSocket sessions.
  • User data stream REST listen key and WebSocket session.
The legacy BinanceAPI wrapper constructs its own context and is not the recommended runtime path for EC2-proxied live operation. Use the main executable flow documented above.

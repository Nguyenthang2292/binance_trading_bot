## Overview

The codebase is architecturally sound — clean modular separation (transport → rest/ws → engine/orders/risk/orchestration), dependency injection, RAII discipline, modern C++23, parameterized SQL, no shell injection, correct HMAC signing, and no secret leakage into logs. **However, it is not production-safe as-is.** The dominant theme is **concurrency and lifetime safety under the real runtime model**: the shared `io_context` runs on **2 threads** ([`src/context.cpp:76-91`](../../src/context.cpp), [`src/main.cpp:829`](../../src/main.cpp)), yet the WS stack, market scanner, and signal engine run callbacks/coroutines on it **without strands or locks**, producing multiple data races and teardown use-after-frees. Two **security** criticals (missing TLS hostname verification → MITM; API secret never zeroized) and a cluster of **financial** criticals (a sizing risk-cap breach, an optimistic free-margin estimate, inconsistent `double`-vs-string money representation, and backtest↔live fill divergence) round out the ship-blockers.

All 15 review units were covered. The first dispatch hit a transient account rate-limit on 5 subagents (`strategy`, `orchestration`, `account`, `backtest`, root files); these were re-dispatched and completed, so there are **no coverage gaps**. The independent v1.0 review reached the same overall verdict from a different decomposition, which raises confidence in the headline conclusions.

| Dimension | Assessment |
|---|---|
| Code quality | **Good.** Modern C++23, RAII, DI, readable. Marred by several silent-failure patterns (`toDouble()→0`, `intField→0`, `config→{}`) that turn bad input into plausible-but-wrong values. |
| Architecture | **Strong.** Layered, interface-driven, testable (mock ports). Main weakness: object lifetime/shutdown is hand-arranged in `main()` rather than centrally owned/drained. |
| Security / data integrity | **At risk.** Missing TLS hostname verification (CR-1) and unzeroized secret (CR-2) are serious; inconsistent money typing (CR-11) threatens accounting integrity. Otherwise solid: parameterized SQL, no shell, HMAC correct, no secret logging. |
| Performance | **Acceptable at current scale, with cliffs.** Hot-path allocations (`KlineCache::snapshot()` deep-copies, per-message `padded_string`/parser allocs) and blocking SQLite on io threads. |
| Concurrency | **Weakest dimension.** Multiple data races and teardown UAFs because the 2-thread io_context is used without strands/locks across WS/scanner/engine. Headline risk. |
| Testing | **Substantial** (per prior module reviews, 460+ tests; strong in catalog/risk/orders). Gaps: concurrency/teardown paths, decimal-edge cases, and backtest↔live parity are largely unguarded. |

## Findings

### 🔴 Critical

#### CR-1 — Missing TLS hostname verification on all HTTPS/WSS connections (MITM)
**File:** [`src/context.cpp:74`](../../src/context.cpp) → [`src/transport/http_session.cpp:86`](../../src/transport/http_session.cpp), [`src/transport/ws_session.cpp:139`](../../src/transport/ws_session.cpp)
`set_verify_mode(ssl::verify_peer)` validates only that the peer certificate chains to a trusted CA — it does **not** verify the certificate was issued for `fapi.binance.com`/`fstream.binance.com`. `SSL_set_tlsext_host_name(...)` sets SNI but performs no verification. A grep across `src/` for `host_name_verification`/`SSL_set1_host`/`set_verify_callback` finds nothing. Any party holding *any* CA-trusted certificate (rogue CA, a domain they own) can MITM the order/market/account streams — forging prices/fills or harvesting the listenKey path. The SOCKS5 proxy path makes this worse.
**Recommendation:** Before handshake, set hostname verification on both streams: `stream.set_verify_callback(ssl::host_name_verification(host))` (Boost ≥ 1.73) or `SSL_set1_host(native_handle, host.c_str())`. Apply to HTTP and WS.

#### CR-2 — API secret stored in plaintext and never zeroized; multiple copies
**File:** [`src/rest/signer.h:19`](../../src/rest/signer.h), [`src/rest/signer.cpp:64-65`](../../src/rest/signer.cpp), [`src/rest/rest_client.cpp:443-445`](../../src/rest/rest_client.cpp)
The signing secret lives in a freely-copyable `std::string` that is never wiped, and exists in ≥3 heap locations (`Signer::m_secretKey`, `ContextConfig::secretKey`, `RestClient::m_cfg`). Any of these can be freed/reallocated without scrubbing and surface in a core dump or swap → full futures-account compromise.
**Recommendation:** Hold the secret in an `OPENSSL_cleanse`-on-destroy buffer, minimize copies (store once, pass by `std::string_view`), and cleanse in destructors. Drop the duplicate copy in `m_cfg`.

#### CR-3 — No 429/418 back-off on signed/order requests (IP-ban risk)
**File:** [`src/rest/rest_client.cpp:497-547`](../../src/rest/rest_client.cpp)
Only `publicGet` inspects results and calls `m_rateLimiter->penalize(...)` on 429 ([`:479-488`](../../src/rest/rest_client.cpp)). The signed helpers (`signedGet/Post/Put/Delete`) just return the error. A 418 ("IP banned", with `Retry-After`) or 429 on an order endpoint triggers no back-off, so the bot can keep hammering the API straight into a multi-day ban — exactly when it most needs to manage open positions. (v1.0 frames the adjacent root cause as per-endpoint rate-limit cost undercounting — CR-20/CR-21/WR-65; both should be fixed together.)
**Recommendation:** In the signed helpers, on `err.code == 429 || err.code == 418` call `penalize(...)` (honoring `Retry-After`) before returning, mirroring `publicGet`; and reserve correct per-endpoint order/IP weights.

#### CR-4 — SignalEngine shared state raced on the 2-thread io_context (no strand)
**File:** [`src/engine/signal_engine.cpp:598-672`](../../src/engine/signal_engine.cpp) (spawn) + [`:1338-1340`](../../src/engine/signal_engine.cpp), [`:1496-1507`](../../src/engine/signal_engine.cpp)
`run()` `co_spawn`s `monitorTimeExit`/`monitorTrailingStops` onto the 2-thread pool alongside the scan loop, none bound to a strand. They concurrently mutate non-atomic, unlocked members: `m_lastOpenDecision`, `m_geminiEvaluationsThisCycle`/`m_geminiCycleGate`, `m_backtestEvaluationsThisCycle`/`m_backtestCycleGate`. Because the Gemini/backtest gates are **fail-closed risk budgets**, a race can either **bypass the budget cap (place more live orders than allowed)** or wedge the gate closed.
**Recommendation:** Bind all `SignalEngine` coroutines to a single `boost::asio::strand` (preferred — serializes the engine), or make the counters atomic and guard the gates/decision with a mutex.

#### CR-5 — WS stack lacks strand/locks: data races across WsSession/WsClient/UserDataStream
**File:** [`src/ws/ws_client.cpp:226-245`](../../src/ws/ws_client.cpp), [`src/ws/user_data_stream.cpp:204-222`](../../src/ws/user_data_stream.cpp), [`src/transport/ws_session.cpp:80-182`](../../src/transport/ws_session.cpp)
With 2 io threads and no strand: `m_subscriptions` (a `std::map`) is iterated on io threads (incl. error paths outside `m_parserMutex`) while `subscribe*`/`unsubscribe` mutate it from caller threads; `m_listenKey`/`m_session` are written by the reconnect coroutine and read by the keepalive coroutine with no mutex (torn `std::string` read → connect against a corrupt key); and `WsSession::send()`/`stop()` post handlers that touch `m_ws`/`m_connected`/`m_connectedAt` and can call `async_write` while a read/reset runs on the other thread (concurrent ops on one Beast stream is UB; a second `send` before the first completes violates Beast's one-write-at-a-time rule).
**Recommendation:** Give each WS object a strand and bind `co_spawn`, every `async_*`, and all posted handlers to it; serialize outbound frames through a single writer queue; snapshot `m_listenKey` into a local before awaiting.

#### CR-6 — WS callback use-after-free on teardown; self-recreating reconnect session
**File:** [`src/ws/ws_client.cpp:207-215`](../../src/ws/ws_client.cpp), [`src/scanner/market_scanner.cpp:378-392`](../../src/scanner/market_scanner.cpp), [`:514-517`](../../src/scanner/market_scanner.cpp)
`WsSession` keeps itself alive via `shared_from_this()` independent of `WsClient`'s lifetime, and `WsClient::disconnect()` only `post`s an async close without joining; the scanner then destroys the `WsClient`/itself (`m_wsClients.clear()`). An in-flight `async_read` completion or error-delivery then fires `m_onMessage`/`m_onDisconnect` into the freed `WsClient`/`MarketScanner` (dereferencing `m_subscriptions`/`m_cache`/`m_onKlineClosed`) → heap UAF. Separately, the reconnect lambda calls `WsClient::connect()`, which `stop()`s and reassigns the very session whose coroutine is running — destroying the owning `shared_ptr` from inside the coroutine and spawning duplicate connections on every reconnect (risking `-1003`/IP bans).
**Recommendation:** Make `WsClient`/`UserDataStream`/`MarketScanner` use `enable_shared_from_this` and capture `weak_ptr` in callbacks (bail if expired), and/or have `WsSession::stop()` synchronously clear its callbacks before returning. Do **not** recreate the session in the reconnect handler — use `WsSession`'s built-in reconnect and let `m_onReconnect` only notify.

#### CR-7 — MarketScanner callback/state data races (`m_onKlineClosed` set after callbacks are live)
**File:** [`src/scanner/market_scanner.cpp:541`](../../src/scanner/market_scanner.cpp) (write) vs [`:384-385`](../../src/scanner/market_scanner.cpp) (read)
`scanner.start()` ([`main.cpp:900`](../../src/main.cpp)) subscribes and connects WS — kline callbacks begin firing on io threads — but `setOnKlineClosed(...)` is only called **later** ([`main.cpp:1407`](../../src/main.cpp)). The io thread reads `m_onKlineClosed` while the main thread move-assigns the `std::function`: a data race that can dispatch through a half-assigned target (crash) or a mid-construction `candleScheduler`. `m_wsClients` and `m_symbolInfo` are likewise mutated by `start()`/`stop()` and read by `symbols()`/`symbolInfo()` from other threads with no mutex.
**Recommendation:** Set the callback before `start()` (before any `connect()`); guard `m_onKlineClosed`/`m_wsClients`/`m_symbolInfo` with a mutex or confine all access to a strand.

#### CR-8 — Unsafe shutdown: fire-and-forget coroutines & scheduler threads outlive their dependencies (UAF)
**File:** [`src/main.cpp:1478-1484`](../../src/main.cpp), [`src/ws/user_data_stream.cpp:151-161`](../../src/ws/user_data_stream.cpp), [`src/orchestration/qlib_state_store.cpp:70-75`](../../src/orchestration/qlib_state_store.cpp)
`main()` calls `signalEngine.stop(); userData.stop(); scanner.stop();` then returns **without draining the io_context**. `UserDataStream::stop()` `co_spawn`s a detached `deleteListenKey` coroutine capturing `this`/`m_rest`; `main` then destroys `userData` while the 2 io threads may not have run that handler yet → UAF on a destroyed stream/REST client (and a missed listenKey delete). The orchestration scheduler `jthread`s and the `QlibStateStore` reload-timer are stopped only implicitly by reverse-declaration-order destruction; `~QlibStateStore` closes the SQLite handle **without holding `m_mutex`** while a timer callback may still dereference it on another thread, and `cancel()` does not wait for an in-flight handler. Subprocess waits ignore the stop token, so shutdown can also hang for the full timeout ([`process_manager.cpp:297-396`](../../src/orchestration/process_manager.cpp)).
**Recommendation:** Add an explicit ordered shutdown in `main()`: request-stop and join scheduler threads, stop the reload loop, run `deleteListenKey` to completion synchronously, then `ctx.ioc().stop()`/join — all while dependent objects are still alive. Guard `m_db` close under the same mutex the callback takes; thread a cancellation signal into subprocess waits.

#### CR-9 — Sizing: min-notional bump can exceed max-notional / available balance (risk-cap breach)
**File:** [`src/engine/sizing_policy.cpp:30-37`](../../src/engine/sizing_policy.cpp)
After flooring quantity to step size, if `quantity * price < minNotional` the code re-rounds **up** (`std::ceil`) and recomputes notional but **never re-checks against `maxNotional`**. When `minNotional` and `maxNotional` are close (or `maxNotional` was the binding cap = `balance * maxPositionNotionalXAvailableBalance`), the final order silently exceeds the per-position risk cap. The caller only logs a config-inversion warning and proceeds ([`signal_engine.cpp:1377-1389`](../../src/engine/signal_engine.cpp)).
**Recommendation:** After the ceil bump, if `quantity * price > maxNotional` (when `maxNotional > 0`), reject (return zero quantity) so the caller skips the trade rather than over-sizing.

#### CR-10 — Free-margin estimate: silent-zero FP fallback, no floor, ignores reserved/maintenance margin
**File:** [`src/account/account_service.cpp:198-206`](../../src/account/account_service.cpp)
`notional = draft.quantity.toDouble() * draft.assumedPrice.toDouble()` — but `DecimalString::toDouble()` returns `0.0` on parse failure ([`decimal_string.cpp:44-50`](../../src/orders/decimal_string.cpp)), so a malformed quantity silently yields `notional=0`, `initialMargin=0`, and reports the **full balance** as free margin. The estimate also subtracts only the *new* order's `notional/leverage` — ignoring maintenance margin and the symbol's leverage-bracket — so it is optimistically biased exactly in the dangerous direction, and is reported as a raw formatted double with no "sufficient/insufficient" flag.
**Recommendation:** Derive notional from the validated `DecimalString` (treat a 0 quantity as an error), clamp/flag insufficiency explicitly, account for bracket/maintenance margin, and prefer the server test-order result over the estimate when both exist.

#### CR-11 — Inconsistent money representation: `OrderUpdateEvent` uses `double` while REST uses `std::string`
**File:** [`src/types/events.h:104-117`](../../src/types/events.h) (also [`:62-67`](../../src/types/events.h) `LiquidationEvent`)
The live order-execution feed models `avgPrice`, `lastFilledPrice/Qty`, `accumulatedFilledQty`, `realizedPnl`, `commission`, `stopPrice` as `double` (populated via `std::stod` in [`user_data_stream.cpp:77-82`](../../src/ws/user_data_stream.cpp)), whereas the parallel REST types `Order`/`UserTrade` ([`trade.h:58-87`](../../src/types/trade.h)) deliberately keep these as `std::string` for exact precision. The same economic quantity therefore has two contradictory representations; accumulating streamed `realizedPnl`/`commission` or comparing a streamed `avgPrice` to a REST string will drift (binary `double` cannot represent decimal ticks). Highest-impact data-integrity issue for accounting. (v1.0 CR-36 flags the same double-money issue at the account-type level; both surfaces need a consistent decimal type.)
**Recommendation:** Represent monetary/quantity fields in `OrderUpdateEvent`/`LiquidationEvent` as `std::string` (or fixed-point) to match the REST types; convert to numeric only at the last moment.

#### CR-12 — Backtest↔live fill divergence undermines the gate that decides live capital
**File:** [`src/backtest/backtest_engine.cpp:221`](../../src/backtest/backtest_engine.cpp),[`:227`](../../src/backtest/backtest_engine.cpp) vs [`src/engine/signal_engine.cpp:1790-1792`](../../src/engine/signal_engine.cpp); [`backtest_engine.cpp:141-234`](../../src/backtest/backtest_engine.cpp); [`indicator_adapters.cpp:387-388`](../../src/backtest/indicator_adapters.cpp)
Three distinct divergences make the simulation unfaithful to live execution: (a) **SL tick-rounding is inverted** — the backtest rounds Long SL *up*/Short SL *down*, but live rounds Long SL *down*/Short SL *up* and shifts by one tick (`stopTickOffset`), so simulated losses are systematically tighter/smaller than live; (b) **same-bar exit-then-reentry** — within one loop iteration a position can exit on bar `i+1` and a new one open at the *open* of that same bar `i+1`, a fill at a price already traded through; (c) **Gartley adapter evaluates at `n-2`** while the engine enters at `i+1`, a two-bar misalignment inconsistent with the other adapters. Since the module exists to gate live trades by backtest performance, these bias the gate. (v1.0 independently flags (a) as WR-11, (b) as WR-9, and (c) as CR-3.)
**Recommendation:** Extract a shared SL/TP quantization helper used by both paths; forbid re-entry on the same bar an exit occurred; align the Gartley evaluation index (`n-1`) with the engine's entry assumption. Add a backtest↔live parity test.

### 🟡 Warning

#### WR-1 — Scientific-notation decimal fails `parse`, leaving an expired position open
**File:** [`src/engine/signal_engine.cpp:42-50`](../../src/engine/signal_engine.cpp)
`toDecimal` uses default `setprecision(16)` formatting, which emits `1e-05` for small quantities; `DecimalString::parse` rejects `e`/`-` and returns `nullopt`, so time-exit close-out gets an empty qty and the position is **not closed**, only logged ([`:2167-2169`](../../src/engine/signal_engine.cpp)).
**Recommendation:** Format with `std::fixed` + explicit precision, or route through `quantityToStepDecimal`.

#### WR-2 — Blocking synchronous SQLite (5 s busy_timeout) on an io_context thread
**File:** [`src/engine/qlib_execution_planner.cpp:346-472`](../../src/engine/qlib_execution_planner.cpp), [`:84`](../../src/engine/qlib_execution_planner.cpp)
`executeMarket` runs synchronous SQLite on the io thread; with `busy_timeout=5000ms` and a 2-thread pool, two concurrent executions can freeze the whole engine, starving WS/order coroutines.
**Recommendation:** Run SQLite work on a dedicated `thread_pool` and post results back (as Gemini already does); lower `busy_timeout`.

#### WR-3 — Qlib partial-fill leaves a position with no SL/TP and over-closes
**File:** [`src/engine/qlib_execution_planner.cpp:436-446`](../../src/engine/qlib_execution_planner.cpp), [`signal_engine.cpp:1672-1683`](../../src/engine/signal_engine.cpp)
If a later slice fails after earlier slices filled, the caller treats it as "entry failed", removes the tracker entry, and `closeByMarket`s the **full** original qty — over-closing relative to the actual fill, with a window where real exposure exists with no protective orders. (v1.0 CR-8/CR-9.)
**Recommendation:** Report cumulative filled qty on partial failure; reconcile/close the exact filled amount and attempt protective orders before surfacing the error.

#### WR-4 — Exposure hard limit bypassed for "deviation-improving" trades
**File:** [`src/engine/exposure_controller.cpp:191-201`](../../src/engine/exposure_controller.cpp)
`if (newDeviation <= currentDeviation) return Allow;` runs **before** the hard-limit block, so a trade that reduces deviation is allowed even if the result still exceeds the hard limit (a large opposite-side order can overshoot through zero). The "hard" limit is not a true ceiling.
**Recommendation:** Cap allowed size so `newDeviation` cannot exceed `hardLimit`, or explicitly document the intended de-risking bypass.

#### WR-5 — Asymmetric reversal guard: `latestDecisionContradicts` only checks the long side
**File:** [`src/engine/qlib_execution_planner.cpp:328-333`](../../src/engine/qlib_execution_planner.cpp)
The contradiction check only blocks when a Buy slice's latest decision flipped away from buy/long; for a Sell slice a reversed (now long/buy) signal does **not** revoke execution. (v1.0 CR-7.)
**Recommendation:** Add the symmetric case for `side == Sell` (`action != "sell" || direction != "short"`).

#### WR-6 — `KlineCache::snapshot()` deep-copies the kline vector on every hot-path call
**File:** [`src/scanner/kline_cache.h:49`](../../src/scanner/kline_cache.h)
Called per work item, per ATR, per Gemini cache-key build (per timeframe), per beta calc, per trailing eval — N symbols × strategies × intervals of full-vector copies each scan cycle.
**Recommendation:** Offer a read API returning `shared_ptr<const vector<Kline>>` or a locked callback for read-only consumers.

#### WR-7 — `pendingReconcile()` is never consumed — orphaned live orders go undetected
**File:** [`src/orders/order_journal.h:37`](../../src/orders/order_journal.h), set at [`normal_order_service.cpp:375-376`](../../src/orders/normal_order_service.cpp)
Every ambiguous network/5xx/timeout placement records `UnknownPendingReconcile`, but no code in `src/` reads it back. "We don't know if this order exists" is written down and never reconciled. (Related to v1.0 CR-8/CR-14.)
**Recommendation:** Confirm an external reconciler consumes `pendingReconcile()` at startup; if none exists, add one. (Operationally the most important warning.)

#### WR-8 — `DurableOrderJournal` constructor swallows path failure (thinks it's durable, can't persist)
**File:** [`src/orders/order_journal.cpp:215-222`](../../src/orders/order_journal.cpp)
On `loadFromFile()` failure the ctor opens a probe `ofstream` and ignores its result, so construction always "succeeds" and `journalIsDurable` stays true; the failure only surfaces per-order. No startup signal that the journal path is unwritable.
**Recommendation:** Probe the path in the ctor and store a hard-fail flag so the service refuses to start (or deterministically rejects every order).

#### WR-9 — Journal load aborts entirely on a single corrupt line
**File:** [`src/orders/order_journal.cpp:48-54`](../../src/orders/order_journal.cpp), [`:306-313`](../../src/orders/order_journal.cpp)
`toInt`/`toInt64` use `std::stoi`/`std::stoll` with no try/catch; one truncated/garbled line (exactly the post-crash scenario a durable journal must survive) throws out of `loadFromFile` → ctor → terminates service construction. (v1.0 WR-52.)
**Recommendation:** Wrap per-line parsing in try/catch and skip unparseable lines.

#### WR-10 — Raw/extra params unvalidated for `closeByMarket`/`stopEntry`/`protection` (latent)
**File:** [`src/orders/order_validator.cpp:174-231`](../../src/orders/order_validator.cpp)
`validateRawParams` is only called for market/limit drafts. These three draft types carry no `raw` map today (so latent), but if one is added, the allowlist that blocks a second `timestamp`/`signature` is bypassed for them. (Related to v1.0 WR-53.)
**Recommendation:** Route every draft's raw params through `validateRawParams`; have the mapper drop blocked keys.

#### WR-11 — Fill-summary conflates entry/exit VWAP and emits understated "Partial" PnL
**File:** [`src/orders/normal_order_service.cpp:680-724`](../../src/orders/normal_order_service.cpp)
`avgEntryPrice` and `avgExitPrice` are both set to the same blended VWAP over all trades regardless of side; and if PnL parses for some trades but not others, a partial sum is emitted as a concrete `realizedPnl` merely labeled `Partial`, which a consumer ignoring `completeness` will undercount.
**Recommendation:** Partition fills by side; suppress `realizedPnl`/commission entirely (leave nullopt) on any parse failure.

#### WR-12 — `batchOrders` silently truncates to 5 (drops orders, e.g. SL legs)
**File:** [`src/rest/rest_client.cpp:1173-1179`](../../src/rest/rest_client.cpp)
`min(5, reqs.size())` caps to Binance's limit but **silently discards** orders 6..N with no error, so the caller believes all were submitted. (v1.0 rates this Critical — CR-17.)
**Recommendation:** Return an error (or partial-result indicator) when `reqs.size() > 5` instead of dropping.

#### WR-13 — Kline `isClosed` derived from local wall-clock vs server `closeTime`
**File:** [`src/rest/rest_client.cpp:344-346`](../../src/rest/rest_client.cpp) (consumed by scanner via `merge`)
`isClosed = closeTime < nowMs` uses `system_clock`; clock skew marks a still-forming candle as closed (or vice-versa). A bot keying decisions off closed candles can act on an unfinished bar; backfilled "closed" bars also flow into the live cache.
**Recommendation:** Derive closed-ness from interval/openTime or a server-time reference; don't fabricate `isClosed` for the last bar.

#### WR-14 — RateLimiter reservations never released on request failure → self-throttling
**File:** [`src/rest/rate_limiter.cpp:93-97`](../../src/rest/rate_limiter.cpp)
`m_reservedWeight/Orders` only reset when the window rolls; a burst of failed/cheap calls makes the reserved tally grow monotonically within a window (`acquire` uses `max(headerUsed, reserved)`), prematurely throttling order placement.
**Recommendation:** Release reservations when the request completes, or decrement once real header usage is known.

#### WR-15 — `RequestGate` cancellation leaves a dead waiter → all later requests hang
**File:** [`src/transport/http_session.cpp:35-72`](../../src/transport/http_session.cpp)
If an awaiting coroutine is cancelled/destroyed while still in `m_waiters`, its stale timer stays at the front; the next `unlock()` cancels a timer nobody waits on and the gate is never handed off → every subsequent request hangs forever. Wakeups also don't re-validate state. (v1.0 rates this Critical — CR-34.)
**Recommendation:** On wakeup, re-check state under the mutex in a loop; remove waiters in their cancellation path. Consider `asio::experimental::channel` / an async mutex.

#### WR-16 — Rate-limit header parser rejects valid values → stale weight → ban risk
**File:** [`src/transport/rate_limit_headers.cpp:23-43`](../../src/transport/rate_limit_headers.cpp)
`from_chars` requires `ptr == end`, so any trailing whitespace/`\r` (not guaranteed trimmed by all proxies) makes the weight update fail silently; the limiter then runs on stale `usedWeight1m`. `int` parse also drops large values.
**Recommendation:** Trim ASCII whitespace before parsing; parse into `long long` and clamp; log at debug when a present header fails to parse.

#### WR-17 — SOCKS5 bound-address length trusted from an untrusted proxy without bound
**File:** [`src/transport/socks5_proxy.h:76-79`](../../src/transport/socks5_proxy.h)
`len[0]` is used to size the follow-up read with no sanity bound; an under-/over-sending proxy desyncs the stream (blocks on the 30 s timer, or corrupts the subsequent TLS handshake) rather than failing cleanly. (Related to v1.0 WR-83.)
**Recommendation:** Treat any framing/length anomaly as a hard, distinct error; keep the connect deadline armed across the tunnel handshake.

#### WR-18 — `intField` has no numeric-string fallback → silent `0` for `orderId`/timestamps
**File:** [`src/ws/ws_parse_helpers.h:25-29`](../../src/ws/ws_parse_helpers.h)
Unlike `doubleField` (which falls back to `std::stoll` on string), `intField` only tries `get(int64_t)`; if Binance sends an integer-valued field as a JSON string, it silently becomes `0`, misrouting order-update reconciliation/timestamps. (v1.0 WR-95.)
**Recommendation:** Give `intField` the same string fallback as `doubleField`.

#### WR-19 — `unsubscribe` is broken: case-folding, no reconnect rebuild, lost frames
**File:** [`src/ws/ws_client.cpp:162-177`](../../src/ws/ws_client.cpp), [`src/transport/ws_session.cpp:80-90`](../../src/transport/ws_session.cpp)
`unsubscribe()` lowercases the *entire* stream name, so it never matches keys whose suffix has uppercase (`@bookTicker`, `@markPrice`, `@forceOrder`, `1M`/`3M` klines) — the `erase` is a no-op and a wrong UNSUBSCRIBE frame is sent. It also doesn't reconcile `buildStreamPath()`, so unsubscribed streams resume after the next reconnect; and `send()` silently drops frames when disconnected.
**Recommendation:** Lowercase only the symbol via one shared key-normalization helper; track the desired stream set as the source of truth and rebuild on reconnect; queue control frames until connected.

#### WR-20 — `UserDataStream::stop()` races the reconnect handler → resurrected stream / leaked listenKey
**File:** [`src/ws/user_data_stream.cpp:151-214`](../../src/ws/user_data_stream.cpp)
A reconnect in flight can run `refreshListenKeyAfterReconnect` after `stop()`, creating a new session/key (resurrecting the stream the caller asked to close, and deleting the wrong key). No "stopping" flag suppresses the reconnect path. (v1.0 WR-94.)
**Recommendation:** Add `std::atomic<bool> m_stopping`, set in `stop()` and checked before any session/key recreation.

#### WR-21 — `cancelBackfill()` blocks an io thread → deadlock risk with the 2-thread pool
**File:** [`src/scanner/market_scanner.cpp:445-450`](../../src/scanner/market_scanner.cpp)
The code requires pool size ≥ 2 so a blocking `state->done.wait()` can be satisfied by another thread, but the deployed pool is exactly 2 and shared by backfill, warmup, WS reads, and REST continuations; a `cancelBackfill().wait()` invoked from an io thread (reachable via `start()`→`stop()`) can wedge both threads. (v1.0 WR-74; v1.0 CR-4 flags the related `use_future().get()` blocking in the kline adapter.)
**Recommendation:** Never block an io thread on a coroutine future — await completion asynchronously (channel/`async_wait`), or do the blocking wait only on a dedicated non-io thread.

#### WR-22 — Non-finite Sortino/UPI (`+inf`) persisted to SQLite (lossy) → breach status unstable across restart
**File:** [`src/risk/risk_metrics.cpp:78-83`](../../src/risk/risk_metrics.cpp), [`src/risk/risk_controller.cpp:121-125`](../../src/risk/risk_controller.cpp)
Zero-downside/zero-drawdown windows set `sortinoRatio`/`upi` to `+inf`; SQLite stores `+Inf` as `NULL` → warm-start reloads `0.0`, so the same data that evaluated `OK` before a restart can evaluate `SOFT/HARD_BREACH` after. `+inf` also pollutes structured logs. (v1.0 WR-71/CR-23 cover the adjacent non-finite handling.)
**Recommendation:** Represent "undefined ratio" with a serialization-safe sentinel (`bool *Defined`); guard `bind_double` against non-finite values; on warm-start, if the last persisted state was a hard breach, seed `HARD_BREACH` rather than re-deriving `OK`.

#### WR-23 — Retention `DELETE` runs on the synchronous recompute/executor path
**File:** [`src/risk/risk_controller.cpp:187-189`](../../src/risk/risk_controller.cpp)
Every recompute issues a `DELETE ... WHERE timestamp_ms < ?` under the DB mutex on the scan executor thread, coupling retention I/O to the safety-metric path; delete+insert isn't transactional. (v1.0 WR-72/IN-3.)
**Recommendation:** Move pruning off the recompute path (separate cadence) and wrap recompute DB work in one transaction; run blocking SQLite off the scan executor.

#### WR-24 — Promotion gate default lets a break-even/under-edge model reach live
**File:** [`src/orchestration/promotion_checker.h:12`](../../src/orchestration/promotion_checker.h), [`promotion_checker.cpp:169-171`](../../src/orchestration/promotion_checker.cpp)
`minMeanNetReturnBps` defaults to `0.0` with a `>=` gate, so a model whose mean **net** return (after *estimated* costs) is exactly 0 — or marginally positive within noise — can be promoted to live. The single most financially-consequential default in the module. Sharpe annualization also overstates with overlapping/serially-correlated outcomes ([`:69-130`](../../src/orchestration/promotion_checker.cpp); v1.0 WR-47).
**Recommendation:** Require a strictly-positive net-return margin (buffer over estimated costs), or require the lower confidence bound > 0; de-overlap/Newey-West-adjust the Sharpe.

#### WR-25 — Uncaught `SQLITE_BUSY`/exception in promotion terminates the candle scheduler thread
**File:** [`src/orchestration/candle_scheduler_thread.cpp:75-104`](../../src/orchestration/candle_scheduler_thread.cpp), [`qlib_state_store.cpp`](../../src/orchestration/qlib_state_store.cpp)
`computeStats` throws on DB error; nothing in `evaluate()`→`processCandle()`→`run()` catches it, and the thread lambda has no handler → the candle scheduler thread dies (or `std::terminate`) on a transient `SQLITE_BUSY` under contention. (Related to v1.0 WR-46.)
**Recommendation:** Wrap `processCandle`/`evaluate` in try/catch within `run()`; log and continue. Set consistent PRAGMAs across all connections.

#### WR-26 — `ProcessManager` global mutex serializes all spawns; `setExecutionMode` holds a mutex across a 5 s transaction
**File:** [`src/orchestration/process_manager.cpp:184`](../../src/orchestration/process_manager.cpp), [`qlib_state_store.cpp:254-348`](../../src/orchestration/qlib_state_store.cpp)
`spawnWithRetry` holds `m_mutex` across every attempt *and* the inter-attempt sleeps, so a multi-minute batch run blocks the latency-sensitive candle pipeline. `setExecutionMode` holds `m_mutex` across `BEGIN IMMEDIATE`+updates+`COMMIT` (up to `busy_timeout`=5 s), stalling `snapshotForAdapter()` on the order-decision hot path. (v1.0 WR-41.)
**Recommendation:** Don't hold the mutex during `spawnOnce`/sleeps or during the SQLite transaction — take it only to publish results/snapshot.

#### WR-27 — Batch scheduler DST/cross-midnight handling can skip or shift a retrain
**File:** [`src/orchestration/batch_scheduler_thread.cpp:296-341`](../../src/orchestration/batch_scheduler_thread.cpp)
`mktime` with `tm_isdst=-1` can shift the wake hour on DST-transition days; combined with `shouldRunToday`'s hour comparison, a scheduled retrain can be missed or run an hour off. (Related to v1.0 WR-40.)
**Recommendation:** Anchor scheduling to UTC or a single civil-time "next instant ≥ now" computation; re-validate the normalized hour after `mktime`.

#### WR-28 — `isDaemonRunning` treats `WAIT_FAILED` as "exited" → possible duplicate watcher
**File:** [`src/orchestration/process_manager.cpp:537-552`](../../src/orchestration/process_manager.cpp)
A transient `WAIT_FAILED` (Windows) erases a still-running daemon from tracking; a second `startDaemon` then spawns a duplicate execution-watcher acting on the same model/DB.
**Recommendation:** Check `GetExitCodeProcess`/`STILL_ACTIVE` explicitly; erase only on confirmed termination.

#### WR-29 — Unvalidated REST bars merged into the shared live `KlineCache`
**File:** [`src/backtest/rest_backfilling_historical_window_provider.cpp:104-113`](../../src/backtest/rest_backfilling_historical_window_provider.cpp)
The provider validates only count and `back().openTime`, then merges into the live `scanner::KlineCache`; no OHLC sanity (finite, high≥max(o,c,l), low≤min, interval spacing). A malformed/hostile response poisons the cache that live decisions read. (v1.0 WR-13/WR-73.)
**Recommendation:** Validate bar integrity (finite, OHLC ordering, spacing == intervalMs) before `merge`.

#### WR-30 — `+inf` Sortino/Sharpe leaks into the combo mean → discards all-winning folds
**File:** [`src/backtest/backtest_engine.cpp:256-293`](../../src/backtest/backtest_engine.cpp), [`backtest_gate_controller.cpp:169-185`](../../src/backtest/backtest_gate_controller.cpp)
Degenerate (no-downside/zero-variance) folds return `+inf`; `meanIS/meanOOS` become `+inf` and the later `isfinite` check then **rejects the entire combo** even though it won every trade. Ratios are also population-denominator and unannualized.
**Recommendation:** Cap degenerate ratios to a finite sentinel (or exclude inf folds from the mean); document the stat conventions.

#### WR-31 — `roundToTick` passes through non-positive prices → short TP silently disabled
**File:** [`src/backtest/backtest_engine.cpp:70-81`](../../src/backtest/backtest_engine.cpp)
The `value <= 0` guard returns the value unrounded; a high-ATR short TP (`entry - tpMult*atr`) can be ≤ 0 and is used directly as an unreachable/negative `tp`, silently disabling the TP leg. (Related to v1.0 WR-12.)
**Recommendation:** Reject (or clamp) the trade when a derived SL/TP is non-positive.

#### WR-32 — Account leverage-missing silently yields no estimate; source-row field parity
**File:** [`src/account/account_service.cpp:190-206`](../../src/account/account_service.cpp), [`mql4_account_adapter.cpp:77-91`](../../src/account/mql4_account_adapter.cpp)
When the symbol has no position row, `leverage` stays 0 and the `if (leverage > 0)` guard skips estimation, returning `Unavailable` with no explanation. Separately, `displayAssetBalance` prefers `/balance` rows over `/account` assets without ensuring field parity, so `accountEquity/Margin/Profit` can read `0.0` defaults. (v1.0 rates the source-row issue Critical — CR-1.)
**Recommendation:** Return an explicit "leverage unknown" note; select source per-field (wallet/available from `/balance`, margin/PnL from `/account`).

#### WR-33 — `BinanceError::message` stores raw upstream HTTP bodies, logged verbatim
**File:** [`src/types/error.h:26`](../../src/types/error.h), [`error.cpp:54-74`](../../src/types/error.cpp)
Non-JSON error bodies are copied verbatim into `message` and emitted by `toString()` into logs with no length cap or sanitization. Not a confirmed secret leak (Binance bodies don't echo keys/signatures), but uncontrolled upstream text in logs.
**Recommendation:** Truncate/strip control chars before storing; document that `message` may contain raw upstream text.

#### WR-34 — Pervasive `double` money with exact-equality risk on exchange filters/positions
**File:** [`src/types/market.h:74-78`](../../src/types/market.h), [`src/types/account.h:24-35`](../../src/types/account.h)
`tickSize`/`stepSize`/`minNotional` and `Position::positionAmt`/`liquidationPrice` are `double`; decimal increments have no exact `double` representation, so tick rounding can land one tick off and `positionAmt == 0.0`/`notional >= minNotional` comparisons are unsafe. (v1.0 CR-36/WR-87.)
**Recommendation:** Store exchange-filter fields as string/fixed-point; use epsilon/integer-tick arithmetic, never `==`/`>=` on raw money doubles.

#### WR-35 — Root: signal handler does non-async-signal-safe work
**File:** [`src/main.cpp:77-80`](../../src/main.cpp)
`signalHandler` calls `Logger::log()` (mutex + allocation + iostream) — not async-signal-safe; a SIGINT/SIGTERM arriving while another thread holds the logger mutex or is in malloc can deadlock or corrupt state during operator shutdown.
**Recommendation:** In the handler, do only `g_running = false;` (atomic store); move the log line to the main thread after the wait loop observes the flag.

#### WR-36 — Root: malformed `config.json` is silently treated as empty → live trading with defaults
**File:** [`src/main.cpp:509-517`](../../src/main.cpp)
A parse error is caught, logged at Warning, and execution proceeds with `config = {}`, so a typo in risk/exposure/gemini config silently runs the bot on default parameters.
**Recommendation:** Distinguish "file absent" (ok → defaults) from "file present but invalid JSON" (fatal → log Error, `return 1`).

#### WR-37 — Root: logger uses `std::localtime` (not thread-safe); `m_minLevel` unsynchronized
**File:** [`src/logger.cpp:46`](../../src/logger.cpp), [`:24`](../../src/logger.cpp),[`:68`](../../src/logger.cpp)
`std::localtime` returns a shared static `std::tm`; concurrent `localtime`/`gmtime` from other code is UB. `m_minLevel` is read/written without the mutex (benign today — set once at startup).
**Recommendation:** Use `localtime_r`/`localtime_s` into a local `tm`; make `m_minLevel` atomic if runtime changes are ever added.

#### WR-38 — `common`: `std::expected` polyfill injected into `namespace std` (UB); `variant<T,E>` unsafe if convertible
**File:** [`src/common/expected_compat.h:13`](../../src/common/expected_compat.h), [`:93`](../../src/common/expected_compat.h)
Defining `unexpected`/`expected`/`bad_expected_access` in `namespace std` is UB; it only "works" because the real `<expected>` is present today. The `std::variant<T,E>` + `holds_alternative<T>` discrimination breaks if `T==E` or `T` is convertible to `E` (latent — all current instantiations use distinct types). (v1.0 WR-23/WR-24.)
**Recommendation:** Move the polyfill into a project namespace with `using` aliases; discriminate on an explicit tag and `static_assert(!is_same_v<T,E>)`.

#### WR-39 — `catalog`: hash→load TOCTOU still open; cross-DLL strategy lifetime unenforced
**File:** [`src/catalog/plugin_loader.cpp:195-201`](../../src/catalog/plugin_loader.cpp), [`strategy_catalog.h:46-50`](../../src/catalog/strategy_catalog.h)
The SHA-256 verify→load window (prior `catalog-v2.0` M2) remains exploitable — it was *documented*, not closed — so an attacker with write access to the confined plugins dir can swap the file between hash and `LoadLibraryExW`/`dlopen`, defeating the allowlist. Separately, the cross-DLL `shared_ptr<IStrategy>` lifetime is safe only by `main()`'s declaration order; nothing in the catalog types enforces "clear the registry before unload". (v1.0 rates these Critical — CR-5/CR-6.)
**Recommendation:** Hold the file open with deny-write share / advisory lock across hash+load (or hash the mapped image); make the strategy lifetime self-enforcing (refcount the owning `PluginHandle` in each strategy's deleter).

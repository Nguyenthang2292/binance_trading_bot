## CRITICAL (must fix)

### C1 — DLL unloaded while `destroyFn` deleter still alive

**Files:** [`plugin_loader.cpp:13-14`](../../src/catalog/plugin_loader.cpp), [`strategy_catalog.cpp:46-48`](../../src/catalog/strategy_catalog.cpp)

`loadAll()` calls `m_handles.clear()` which destructs all `PluginHandle` objects and triggers `FreeLibrary`/`dlclose`. If `initialize()` is called a second time while `StrategyRegistry` still holds `shared_ptr<IStrategy>` objects from the first load, those pointers' custom deleters (`destroyFn`) and vtables point into now-unloaded memory.

**Result:** USE AFTER FREE — guaranteed undefined behavior on the second call to `initialize()`.

**Fix:** Add `m_registry.clear()` before `m_loader.loadAll()` inside `initialize()`. This drops all strategy `shared_ptr`s (calling their `destroyFn` while the DLLs are still mapped) before `loadAll()` unloads them.

```cpp
// strategy_catalog.cpp — initialize()
m_info.clear();
m_registry.clear();   // ← ADD: drop shared_ptrs before DLLs are unloaded
const auto pluginResults = m_loader.loadAll();
```

---

### C2 — Relative `pluginsDir` enables DLL hijacking

**Files:** [`plugin_loader.h:25`](../../src/catalog/plugin_loader.h), [`strategy_catalog.h:18`](../../src/catalog/strategy_catalog.h)

Default `pluginsDir = "plugins"` resolves against the process CWD, not the binary's directory. On Windows, `LoadLibraryA` also searches `PATH` and `CWD` for transitive DLL dependencies. An attacker who controls the CWD (misconfigured service, symlink, `CreateProcess` with attacker-chosen working directory) can load arbitrary DLLs.

**Fix:** Resolve `pluginsDir` to an absolute canonical path at startup, anchored to the binary's own directory (`argv[0]` parent), not the CWD.

## HIGH (should fix)

### H1 — Missing `RTLD_LOCAL` on `dlopen`

**File:** [`plugin_handle.cpp:39`](../../src/catalog/plugin_handle.cpp)

```cpp
void* rawHandle = dlopen(dllPath.string().c_str(), RTLD_NOW);
```

Default mode is `RTLD_GLOBAL` — all symbols from loaded plugins go into the global namespace. Two plugins exporting a symbol with the same name (e.g., both have a `createStrategy` C function) will silently resolve against each other.

**Fix:** `RTLD_NOW | RTLD_LOCAL`

---

### H2 — `LoadLibraryA` drops Unicode paths

**File:** [`plugin_handle.cpp:36`](../../src/catalog/plugin_handle.cpp)

`dllPath.string()` returns an ANSI-encoded narrow string (system code page). Paths containing characters outside the active code page are silently corrupted.

**Fix:** `LoadLibraryW(dllPath.wstring().c_str())`

---

### H3 — `LoadLibraryA` instead of `LoadLibraryExA` with safe search flags

**File:** [`plugin_handle.cpp:36`](../../src/catalog/plugin_handle.cpp)

Even with an absolute path, the loaded DLL's transitive dependencies still use the standard search order (CWD, PATH). This is a vector for dependency hijacking.

**Fix:**

```cpp
HMODULE module = LoadLibraryExA(
    dllPath.string().c_str(), nullptr,
    LOAD_LIBRARY_SEARCH_DLL_LOAD_DIR | LOAD_LIBRARY_SEARCH_SYSTEM32);
```

### H4 — Public mutable function pointer members

**File:** [`plugin_handle.h:36-39`](../../src/catalog/plugin_handle.h)

`createFn`, `destroyFn`, `typeFn`, `versionFn` are public non-const. Any code with a `PluginHandle&` can overwrite them after construction. Security-sensitive for a plugin system handling external code.

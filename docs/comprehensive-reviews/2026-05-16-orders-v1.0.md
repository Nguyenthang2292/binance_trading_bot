## Module Overview

The `src/orders` module handles all order lifecycle operations against the Binance Futures REST API:

- `algo_order_service.h/.cpp` — Stop-entry and protection (TP/SL) algo orders
- `normal_order_service.h/.cpp` — Market, limit, close-by-market, batch orders
- `order_validator.h/.cpp` — Pre-placement validation
- `order_mapper.h/.cpp` — Draft → REST request mapping
- `order_journal.h/.cpp` / `order_id_generator.h/.cpp` — Journaling and ID management
- `order_service_utils.h` — Shared detail utilities (error mapping, state strings, etc.)
- `mql4_adapter.h/.cpp` — MQL4 bridge adapter
- `rest_client_adapter.h/.cpp` — HTTP client abstraction

---

## Change Reviewed

Both `algo_order_service.cpp` and `normal_order_service.cpp` were modified to add metadata-enriched placement logging:

1. **New `extractTimeframe()` helper** (both files, anonymous namespace) — parses `"tf=<value>"` from the start of `metadata->comment`
2. **`logPlacement()` extended** — optional third parameter `const std::optional<OrderMetadata>& metadata = std::nullopt`
3. **Two new log fields** — `strategy=` (from `metadata->strategyTag`) and `tf=` (via `extractTimeframe`) appended to every placement log line
4. **All call sites updated** — all `logPlacement()` calls in both files now pass `prepared->metadata`

## Issues Found and Scored

### Sub-threshold findings (informational)

#### [SCORE 75] `extractTimeframe()` duplicated in both translation units

**Files:** `algo_order_service.cpp:19–33`, `normal_order_service.cpp:52–66`

The function is byte-for-byte identical in both anonymous namespaces. The project already has `src/orders/order_service_utils.h` (`namespace orders::detail`) which both files already import via `using orders::detail::...` declarations. A future logic change (e.g., changing the `tf=` prefix format) must be applied in two places or they will silently diverge.

**Why not blocking:** No functional bug today; compiler catches no ODR violation. Fix is straightforward if done before the `comment` parsing logic grows.

**Suggested fix:** Move to `order_service_utils.h` under `namespace orders::detail`:

```cpp
inline std::optional<std::string> extractTimeframe(const std::optional<OrderMetadata>& metadata) { ... }
```

---

#### [SCORE 65] Batch-level validation failure in `batchNormal` drops metadata

**File:** `normal_order_service.cpp:765`

```cpp
const auto batchValidation = m_validator.validateBatch(drafts);
if (batchValidation.hasErrors()) {
    for (size_t i = 0; i < drafts.size(); ++i) {
        NormalPlacementResult item;
        ...
        logPlacement(item, startedAt);   // no metadata — strategy/tf absent from log
    }
}
```

Every other `logPlacement` call in the same function passes `prepared->metadata` (lines 799, 807, 831, 847, 859, 869). In the batch validation failure path no `PreparedPlacement` exists yet, but metadata IS available from `drafts[i]` via `std::visit`. The omission means batch validation rejections are never attributed to their strategy or timeframe in logs.

**Why not blocking:** Not a functional defect (batch is still rejected correctly); requires `std::visit` to fix which is non-trivial for a failure path.

**Suggested fix (option A):** Pass explicit `std::nullopt` with a comment:

```cpp
logPlacement(item, startedAt);  // batch-level rejection: no PreparedPlacement exists yet
```

**Suggested fix (option B):** Extract metadata from draft variant:

```cpp
const auto draftMeta = std::visit([](const auto& d) { return d.metadata; }, drafts[i]);
logPlacement(item, startedAt, draftMeta);
```

---

#### [SCORE 65] `comment` field overloaded for structured `tf=` data

**File:** `src/orders/order_common.h` (struct definition), both `.cpp` files (parsing)

`OrderMetadata` already has a typed dedicated field `strategyTag`. Encoding the timeframe by parsing `"tf=<value>"` from the free-form `comment` field is inconsistent with that pattern and undocumented. The `mql4_adapter` also carries `OrderMetadata`; MQL4 strategies conventionally use `comment` as a descriptive string, creating a silent collision risk.

**Why not blocking:** Current producers (`signal_engine.cpp`) always write `"tf=<value> ..."` as the comment, so parsing is correct today. No crash, no misbehavior.

**Suggested fix:** Add `std::optional<std::string> timeframe` to `OrderMetadata` (parallel to `strategyTag`) and log it directly, eliminating the parse step.

---

#### [SCORE 25] `starts_with("tf=")` vs `find("tf=")` — false positive

One agent suggested using `find` instead of `starts_with`. This is a false positive. The format contract (established in `signal_engine.cpp`) always places `tf=` at the start of the comment string: `"tf=<interval> reason=<reason>"`. The `starts_with` check is intentionally correct for this format.

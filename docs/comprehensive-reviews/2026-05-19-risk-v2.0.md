## Critical (1)

### CR-1: Risk status is not restored on restart — persisted `HARD_BREACH` is silently dropped ✅ DONE

**Files:** [`risk_controller.cpp:108-126`](../../src/risk/risk_controller.cpp), [`risk_db.cpp:220-258`](../../src/risk/risk_db.cpp), [`config.json:145`](../../config.json)

`RiskController::recomputeMetrics` writes a fresh row to `risk_metrics_cache` every recompute via `m_db.insertMetrics(result)`. `RiskDb::getLatestMetrics` exists precisely to read that row back — but **nothing in production ever calls it**. A grep confirms `getLatestMetrics` is referenced only in `test_risk_db.cpp`; `RiskController` does not call it from its constructor or anywhere else.

Consequence — the restart sequence:

1. The bot hits `HARD_BREACH`; `m_status = HARD_BREACH` is persisted to `risk_metrics_cache`.
2. The process restarts. `RiskController` is constructed with default member state: `m_status{RiskStatus::OK}`, `m_latest{}` (`valid == false`), `m_lastComputeMs{0}`.
3. The first scan cycle runs the per-symbol loop. Each symbol calls `canOpenPosition()` ([`signal_engine.cpp:713`](../../src/engine/signal_engine.cpp)) **before** `maybeRecompute` runs ([`signal_engine.cpp:467`](../../src/engine/signal_engine.cpp), after the loop):

```cpp
bool RiskController::canOpenPosition() const {
    if (!m_config.enabled) return true;
    std::shared_lock lock(m_mutex);
    if (m_status == RiskStatus::HARD_BREACH) return false;   // m_status is OK after restart
    if (!m_latest.isValid()) {
        return m_config.missingDataMode == RiskMissingDataMode::Open;  // prod config: "open" → true
    }
    return true;
}
```

With the production config (`missing_data_mode: "open"`), `canOpenPosition()` returns **true** for the entire first scan cycle. New positions can be opened despite a persisted `HARD_BREACH`. The breach is only re-derived when `maybeRecompute` → `recomputeMetrics` runs at the *end* of that first cycle.

This is a genuine risk-gate bypass. The window is bounded (it self-heals after the first recompute) and the data needed to prevent it is already on disk — but `missing_data_mode` was designed for *genuinely absent* history (fresh install), not for *known, persisted* breaches that simply have not been loaded. Treating "computed and breached" as "no data" is the defect.

**Fix — warm-start in the constructor:**

```cpp
RiskController::RiskController(RiskDb& db, EquityCurve& curve, RiskMetrics metrics, RiskConfig config)
    : m_db(db), m_curve(curve), m_metrics(std::move(metrics)), m_config(std::move(config)) {
    if (auto cached = m_db.getLatestMetrics("rolling", basisString())) {
        m_latest = *cached;
        m_status = evaluate(m_latest);
        m_lastComputeMs = m_latest.computedAtMs;
    }
}
```

Staleness is self-correcting: `m_lastComputeMs` is set from the cached `computedAtMs`, so if the cache is older than `metricsComputeIntervalMinutes`, the interval gate in `maybeRecompute` triggers an immediate fresh recompute on the first scan; if it is recent, the warm-started status holds until then. A stale persisted `HARD_BREACH` blocks trading until the recompute refreshes it — the correct fail-safe direction. (If warm-start is wired up, also address IN-3 first.)

If the metrics cache is intentionally not meant to be read back, then `getLatestMetrics` is dead code and should be deleted — but given this defect, restoring it is the right call.

## Warning (4)

### WR-1: Recompute duration is never logged — violates the design's explicit precondition for synchronous recompute ✅ DONE

**Files:** [`risk_controller.cpp:185-202`](../../src/risk/risk_controller.cpp), [`risk_controller.cpp:224-248`](../../src/risk/risk_controller.cpp)

`maybeRecompute` is declared `boost::asio::awaitable<void>` but contains no `co_await` of an async operation — it calls `recomputeMetrics` synchronously, which performs blocking SQLite I/O (`deleteEquityPointsOlderThan`, `getByTimeRange`, `insertMetrics`) plus CPU work, then `co_return`s. It is `co_await`-ed from the scan coroutine ([`signal_engine.cpp:467`](../../src/engine/signal_engine.cpp)), so it blocks that executor thread for the full recompute.

This synchronous design was a **deliberate, documented v1 tradeoff** (design doc D8: "avoid spawning a thread in the first version; recompute once per hour is enough") — *conditioned* on one mitigation: the design states the recompute "phải log duration" (must log its duration) so operators have a signal for when to upgrade to a worker thread/strand.

`recomputeMetrics` and `logMetrics` log the metric *values* and status, but **never the wall-clock duration**. The safety valve the design relied on is absent: at current data scale (~8.7k rows/year) the block is sub-millisecond and harmless, but there is no instrumentation to detect the day it stops being harmless (e.g., a much shorter `sample_interval_minutes`, or multi-year accumulation).

**Fix:** wrap the recompute in a `std::chrono::steady_clock` measurement and include `duration_ms=` in the `logMetrics` line.

### WR-2: `evaluate()` gates UPI checks on `isValid()` but not the drawdown checks ✅ DONE

**File:** [`risk_controller.cpp:204-222`](../../src/risk/risk_controller.cpp)

```cpp
if (metrics.maxDrawdown < hardDrawdownLimit) return RiskStatus::HARD_BREACH;          // NOT gated
if (metrics.isValid() && metrics.upi < m_config.hardMinUpi) return RiskStatus::HARD_BREACH;  // gated
if (metrics.maxDrawdown < softDrawdownLimit) return RiskStatus::SOFT_BREACH;          // NOT gated
if (metrics.isValid() && metrics.upi < m_config.softMinUpi) return RiskStatus::SOFT_BREACH;  // gated
```

The drawdown comparisons run unconditionally. This is safe **today** only because of an implicit, unenforced invariant: `RiskMetrics::compute` returns early on every invalid path *before* assigning `maxDrawdown`, so an invalid result always carries `maxDrawdown == 0.0`, which never breaches a negative threshold. Nothing enforces or documents that invariant. If a future change reorders `compute` so `maxDrawdown` is assigned before a validity gate, a drawdown breach would fire on garbage data.

The same "invariant enforced at a single entry point" pattern appears with `RiskConfig`: cross-field validation lives only in `fromJson`, so a directly-constructed `RiskConfig` (as test helpers and any future code do) bypasses every threshold check.

**Fix:** make the gating explicit and symmetric — `if (!metrics.isValid()) return RiskStatus::OK;` as the first line of `evaluate()`. Optionally move `RiskConfig` validation into a constructor or a `validate()` method so all construction paths are covered.

### WR-3: No RAII for `sqlite3_stmt` — prepared statements leak on exception ✅ DONE

**File:** [`risk_db.cpp:132-183`](../../src/risk/risk_db.cpp) (`getByYear`, `getByTimeRange`; pattern repeats in `insertEquityPoint`, `insertMetrics`, `getLatestMetrics`, `deleteEquityPointsOlderThan`)

Every DB method follows `prepare → … → finalize` with the `finalize` as a bare statement. In the row-scanning methods, the loop body throws plausibly:

```cpp
while (sqlite3_step(stmt) == SQLITE_ROW) {
    EquityPoint p;
    p.source = columnTextOrEmpty(stmt, 4);  // std::string alloc — can throw bad_alloc
    p.basis  = columnTextOrEmpty(stmt, 5);
    out.push_back(std::move(p));             // vector growth — can throw bad_alloc
}
sqlite3_finalize(stmt);                      // skipped if the loop throws
```

On `std::bad_alloc` (realistic for a large result set) the function exits via exception and `sqlite3_finalize` is skipped. The prepared statement leaks, and a leaked statement keeps resources pinned on the long-lived connection. The `std::lock_guard` releases correctly; only the `sqlite3_stmt*` is unmanaged.

**Fix:** wrap the handle in RAII, e.g. `std::unique_ptr<sqlite3_stmt, decltype(&sqlite3_finalize)>`, or a small `Statement` guard. This also lets the `step`-failure paths drop their manual `finalize`-before-`throw`.

### WR-4: Controller test coverage diverges from the design's stated test matrix ✅ DONE

**File:** [`tests/test_risk_controller.cpp`](../../tests/test_risk_controller.cpp)

The risk module has 14 unit tests across 4 files; `RiskMetrics` is well covered, including a real S&P 500 1992–2019 fixture validated against published Ulcer Index / max-drawdown ground truth — strong numeric coverage. But controller-level coverage is thinner than the design doc's own test plan requires. Missing:

- **`failure_mode` open vs closed** — the exception → `HARD_BREACH` fail-closed path in `maybeRecompute` ([`risk_controller.cpp:154-170`](../../src/risk/risk_controller.cpp)) is untested in both modes. This is core safety logic.
- **`maybeRecompute` interval gating** — no test that a second call within `metricsComputeIntervalMinutes` is skipped, or that the coroutine path works at all (all controller tests call `recomputeMetrics` directly).
- **Equity-basis selection** — `selectEquity` margin-vs-wallet branching ([`risk_controller.cpp:255-260`](../../src/risk/risk_controller.cpp)) is untested.
- **`SOFT_BREACH` and UPI-breach paths** — only the hard drawdown breach is exercised; soft drawdown and both UPI tiers are not.
- **`extractYear`** — the C++ date logic in `EquityCurve` is never exercised (DB tests set `year` explicitly).

**Fix:** add controller tests for the bullets above, prioritizing `failure_mode` (fail-closed correctness) and the equity-basis branch.

## Info (4)

### IN-1: `EquityCurve::record()` silently drops invalid equity points ✅ DONE

**File:** [`equity_curve.cpp:25-36`](../../src/risk/equity_curve.cpp)

```cpp
void EquityCurve::record(double equity, int64_t timestampMs, std::string_view source, std::string_view basis) {
    if (equity <= 0.0 || timestampMs <= 0 || basis.empty()) {
        return;  // silent
    }
    ...
}
```

The guard is correct, but the drop is silent. If an `AccountSnapshot` ever yields zero/negative equity (un-populated snapshot, transient API failure), points vanish from the curve with no diagnostic trail; `dataPoints` shrinks and metrics may silently flip to `invalid`. Recommend a `LogLevel::Debug`/`Warning` line on drop, including which field failed.

### IN-2: Leftover heap allocation in `main.cpp` after the WR-5 fix ✅ DONE

**File:** [`main.cpp:829-837`](../../src/main.cpp)

`riskMetrics` is a `std::unique_ptr<engine::RiskMetrics>`, but since WR-5 the `RiskController` constructor takes `RiskMetrics` **by value** and copies it. After construction the heap `RiskMetrics` is dead weight. `RiskMetrics` is a three-scalar value type — the `make_unique` is unnecessary; construct it on the stack (or inline into the constructor call) and drop the `unique_ptr`.

### IN-3: `risk_metrics_cache` non-finite persistence is unverified ✅ DONE

**Files:** [`risk_metrics.cpp:78-83`](../../src/risk/risk_metrics.cpp), [`risk_db.cpp:211-212`](../../src/risk/risk_db.cpp), [`tests/test_risk_db.cpp`](../../tests/test_risk_db.cpp)

The WR-2 fix made `sortinoRatio` and `upi` legitimately `+infinity` when there are no downside returns/drawdowns. `insertMetrics` binds those via `sqlite3_bind_double`. SQLite only converts `NaN` to `NULL` (via `sqlite3VdbeMemSetDouble`); `±infinity` is stored and retrieved as a valid IEEE 754 8-byte `REAL` and round-trips correctly — no code change to `insertMetrics` or `getLatestMetrics` is needed.

Verified by test `PreservesInfiniteMetricsAcrossCacheRoundTrip` in `tests/test_risk_db.cpp`: inserts a `RiskMetricsResult` with `sortinoRatio = +inf` and `upi = +inf`, reads it back via `getLatestMetrics`, and asserts both values are still `+inf`. The warm-start path introduced by CR-1 will correctly restore `HARD_BREACH`/`SOFT_BREACH` status based on finite metrics and will preserve `+infinity` for the UPI/sortino fields when they were never breached.

### IN-4: `recomputeMetrics` is public and bypasses fail-closed handling ✅ DONE

**Files:** [`risk_controller.h:27`](../../src/risk/risk_controller.h), [`risk_controller.cpp:142-202`](../../src/risk/risk_controller.cpp)

The exception → `HARD_BREACH` fail-closed logic lives in `maybeRecompute`, not in `recomputeMetrics`. `recomputeMetrics` is public (needed for tests) and, if called directly, propagates exceptions raw with no fail-closed fallback. Low impact — only tests call it directly today — but the safety policy belongs with the operation it protects. Consider folding the try/catch into `recomputeMetrics`, or documenting that direct callers must handle failure themselves.

Minor, not separately numbered: `deleteEquityPointsOlderThan` filters on a bare `timestamp_ms` predicate, which neither composite index (`idx_ep_year_ts`, `idx_ep_basis_ts` — both have `timestamp_ms` as the *second* column) can serve, so the hourly prune full-scans `equity_points`. Negligible at current volume; add a dedicated `idx_ep_ts` if `sample_interval_minutes` is ever shortened. Also consider marking `m_config` as `const` to compiler-enforce the immutability that `canOpenPosition`'s lock-free read already assumes.

## Success Criteria Assessment

- **Security:** No injection surface (fully parameterized SQL), null-safe column reads, no secrets. Fail-closed semantics correct.
- **Performance:** No computation in the `canOpenPosition` hot path; recompute is bounded and runs hourly off the per-symbol loop. WR-1 is the only instrumentation gap.
- **Architecture:** Layering and the `IRiskPort` abstraction are sound. CR-1 is an *unfinished* design path (`getLatestMetrics` written but never consumed), not a structural flaw.
- **Testing:** Numeric correctness is well covered with external ground truth; controller behavior coverage has gaps (WR-4).
- **Documentation:** Sign conventions and the C++ standard are now explicit. The archived design doc has drifted from the implementation (WR-1) — acceptable since it is archived, but the D8 contract still applies.

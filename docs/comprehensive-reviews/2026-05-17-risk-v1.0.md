## Overview

The risk module provides equity tracking, statistical risk metrics computation, and position-gate enforcement. It is composed of four layered classes:

| Class | Responsibility |
|---|---|
| `RiskDb` | SQLite persistence for equity points and metrics cache |
| `EquityCurve` | Records equity snapshots keyed by source and basis |
| `RiskMetrics` | Computes Sharpe, Sortino, UPI, max drawdown, Ulcer Index |
| `RiskController` | Orchestrates the above, exposes `IRiskPort` interface |

The architecture is clean, the DB layer uses parameterized statements throughout (no SQL injection risk), and the fail-closed behavior is implemented correctly. Below are findings grouped by severity.

## Critical (2)

### CR-1: Null pointer dereference from `sqlite3_column_text` ✅ DONE

**Files:** [`risk_db.cpp:144-145`](../../src/risk/risk_db.cpp), lines 167–168, 236–237, 239

`sqlite3_column_text` returns `nullptr` for SQL NULL values. The result is directly assigned to `std::string` without a null check, which is undefined behavior.

```cpp
// Current (dangerous)
p.source = reinterpret_cast<const char*>(sqlite3_column_text(stmt, 4));
p.basis  = reinterpret_cast<const char*>(sqlite3_column_text(stmt, 5));
```

All columns are `NOT NULL` in the schema, but database corruption or future schema changes can produce NULLs. A single bad row would crash the process.

**Fix:**
```cpp
auto safeText = [](sqlite3_stmt* s, int col) -> std::string {
    const char* raw = reinterpret_cast<const char*>(sqlite3_column_text(s, col));
    return raw ? raw : "";
};
p.source = safeText(stmt, 4);
p.basis  = safeText(stmt, 5);
```

Affects: `getByYear`, `getByTimeRange`, `getLatestMetrics` — fix all four query sites.

### CR-2: UPI threshold validation missing — `hardMinUpi` vs `softMinUpi` not cross-validated ✅ DONE

**File:** [`risk_controller.cpp:97-103`](../../src/risk/risk_controller.cpp)

Drawdown thresholds are validated against each other (`softMaxDrawdown <= hardMaxDrawdown`), but UPI thresholds are not:

```cpp
// Drawdown: validated ✓
if (cfg.softMaxDrawdown > cfg.hardMaxDrawdown) { throw ...; }

// UPI: NOT validated ✗
// cfg.softMinUpi and cfg.hardMinUpi are accepted as-is
```

If `hardMinUpi > softMinUpi`, the hard breach is triggered first — correct. But if `hardMinUpi < softMinUpi` (inverted), soft breach can never fire: the hard condition fires first. This silently disables the soft-breach tier for UPI.

**Fix — add to `RiskConfig::fromJson`:**
```cpp
if (cfg.hardMinUpi > cfg.softMinUpi) {
    throw std::invalid_argument(
        "risk_analytics.hard_min_upi must be <= soft_min_upi");
}
```

## Warning (5)

### WR-1: `maxDrawdown` sign convention is implicit and fragile ✅ DONE

**File:** [`risk_controller.cpp:204-214`](../../src/risk/risk_controller.cpp)

`computeMaxDrawdown` returns a negative value (e.g., `-0.35` for a 35% drawdown). The evaluation logic negates the config threshold:

```cpp
if (metrics.maxDrawdown < -m_config.hardMaxDrawdown) { ... }  // -0.40 < -0.35 ✓
```

This is correct but fragile. If any caller assigns `maxDrawdown` differently (e.g., via deserialized DB data where the sign is ambiguous), the comparison silently inverts. The struct field name `maxDrawdown` with a positive config threshold `hardMaxDrawdown` creates a sign contract that is not enforced anywhere.

**Recommendation:** Document the sign convention explicitly on `RiskMetricsResult::maxDrawdown` or store it as a positive magnitude and negate at comparison sites consistently.

### WR-2: Sortino sentinel value `99.0` can mask UPI anomalies ✅ DONE

**File:** [`risk_metrics.cpp:78-79`](../../src/risk/risk_metrics.cpp)

```cpp
out.sortinoRatio = out.stdDevDownside > 0.0
    ? (out.excessReturn / out.stdDevDownside)
    : kSentinelRatio;  // 99.0
out.upi = out.ulcerIndex > 0.0
    ? (out.excessReturn / out.ulcerIndex)
    : kSentinelRatio;  // 99.0
```

When there are no negative returns or drawdowns, both ratios are set to `99.0`. In `evaluate()`, `metrics.upi < m_config.hardMinUpi` with `hardMinUpi = -1.0` would always pass (99 > -1), which is the intended behavior. However:

- The sentinel is persisted to the DB as `99.0`, making historical queries indistinguishable from genuinely high-performing periods.
- If `hardMinUpi` is set high (e.g., `2.0` to require strong performance), the sentinel bypasses the check.

**Recommendation:** Use `std::numeric_limits<double>::infinity()` as the sentinel, or store a separate `bool sortinoValid` / `bool upiValid` flag in `RiskMetricsResult`.

### WR-3: No retention policy on `equity_points` table ✅ DONE

**File:** [`risk_db.cpp:63-104`](../../src/risk/risk_db.cpp)

With `recordPeriodic` firing every 60 minutes, the table accumulates ~8,760 rows/year minimum. There is no pruning, no `LIMIT` on inserts, and no archival mechanism. Over multiple years, queries that scan by `basis + timestamp_ms` will degrade proportionally.

**Recommendation:** Add a periodic cleanup that deletes rows older than `controlLookbackDays * 2` (keeping a safety margin beyond the rolling window), or implement row-count-based capping.

---

### WR-4: `sampleEquity` copies `EquityPoint` structs (including strings) per bucket ✅ DONE

**File:** [`risk_metrics.cpp:198-225`](../../src/risk/risk_metrics.cpp)

```cpp
std::unordered_map<int64_t, EquityPoint> byBucket;
```

Each bucket stores a full `EquityPoint` copy including two `std::string` fields (`source`, `basis`). For a 365-day window at 60-minute sampling this is ~8,760 potential copies. At scale or with shorter intervals, this becomes measurable.

**Recommendation:** Since the output only uses `timestampMs` and `equity`, either store `SampledEquityPoint` directly in the map (dropping the strings at bucket-insertion time), or store an index into `raw`.

---

### WR-5: Dangling reference risk — `m_metrics` stored as `const RiskMetrics&` ✅ DONE

**File:** [`risk_controller.h:43`](../../src/risk/risk_controller.h)

```cpp
const RiskMetrics& m_metrics;
```

`RiskController` holds a reference to `RiskMetrics` but does not own it. If the owner destroys `RiskMetrics` before `RiskController`, all calls to `recomputeMetrics` use a dangling reference. There is no static enforcement of the lifetime contract.

**Recommendation:** Take ownership (`RiskMetrics m_metrics;`) or use `std::shared_ptr<const RiskMetrics>` if sharing is required.

## Info (4)

### IN-1: `HARD_BREACH` logged at `Warning` level instead of `Error` ✅ DONE

**File:** [`risk_controller.cpp:225-227`](../../src/risk/risk_controller.cpp)

```cpp
} else if (status == RiskStatus::HARD_BREACH) {
    statusText = "HARD_BREACH";
    level = LogLevel::Warning;  // should be Error
}
```

A hard breach halts all new position opens. Logging it at `Warning` means it may be filtered out in production environments configured to suppress warnings. It should log at `Error` or higher.

---

### IN-2: `quoteString` creates an unnecessary intermediate `std::string` ✅ DONE

**File:** [`risk_controller.cpp:19-23`](../../src/risk/risk_controller.cpp)

```cpp
out << std::quoted(std::string(value));  // copies string_view → string
```

In C++20, `std::quoted` accepts `std::string_view` directly. The intermediate construction is avoidable:

```cpp
out << std::quoted(value);  // C++20, no copy
```

### IN-3: `extractYear` uses C++20 `std::chrono::year_month_day` ✅ DONE

**File:** [`equity_curve.cpp:38-43`](../../src/risk/equity_curve.cpp)

`std::chrono::year_month_day` is a C++20 addition. This is fine if the project targets C++20, but should be confirmed in `CMakeLists.txt`. If the project ever needs to support C++17, this will break silently on some compilers.

## Overview

The engine orchestrates the open→manage→close lifecycle of a trade: `runScanCycle` builds a randomized (symbol, strategy, interval) work queue, evaluates each candidate through a chain of gates (sizing → order cap → exposure → risk → gemini → place orders), then asynchronously monitors time-exit, trailing-stop and loss-manager (break-even / DCA) work in independent coroutines.

The dominant design tensions are:

1. **Per-trade random leverage between 2× and 20×** ([`signal_engine.cpp:161-168`](../../src/engine/signal_engine.cpp)) — a single position's margin usage, liquidation distance, and ROI exposure swing 10× across calls. There is no log of the chosen leverage at the planning step, no config gate, no determinism for backtesting. This is the most surprising design choice in the module and should be either justified inline or removed.
2. **`WorkQueue::build` evaluates only one strategy per symbol per cycle** ([`work_queue.cpp:37-50`](../../src/engine/work_queue.cpp)) — with N strategies and M symbols, only M (strategy, symbol) pairs are scored per cycle instead of N×M. The behavior is documented inline as "probabilistically fair over time via shuffle", but the warning log only fires when `strategies > symbols`. With M=100 symbols and N=3 strategies, two-thirds of strategy/symbol combinations are silently skipped every cycle.
3. **`openPosition` accepts silent stop-loss placement failure** ([`signal_engine.cpp:1071-1099`](../../src/engine/signal_engine.cpp)) — if `protection()` rejects, the call logs a warning and continues with the position open, no SL on the exchange. Combined with `placeStopLoss=false` being the production default in `config.json`, the actual safety net depends on `monitorTimeExit` + trailing stops alone.
4. **`signal_engine.cpp` is 1517 lines in a single file** with `openPosition` alone spanning ~500 lines. The function mixes sizing, gates, leverage selection, market+limit+protection placement, tracker reservation, and error recovery. Splitting into smaller named operations would help correctness review.

Other concerns: a thread-per-evaluation pattern in `evaluateGeminiNonBlocking` (`std::async` + 25ms polling timer), busy-polling in `monitorTrailingStops`, and a placement→cancel race that can leave a position open with no protection if either `closeByMarket` or the SL cancel fails during time-exit.

| Dimension | Assessment |
|---|---|
| Code quality | Mixed. Small modules (sizing, price_filter, work_queue) are tight and clear. `signal_engine.cpp` is overlong and mixes concerns. |
| Architecture | Good port abstractions. Strategy/Signal flow is testable. Cycle gate state machine (Gemini) is reasonable. |
| Security / data integrity | Good — no SQL or shell injection surface in this module. Random `std::random_device{}` + thread-local `std::mt19937` is fine for non-crypto uses. |
| Performance | One thread-per-evaluation pattern (gemini), busy-polling in trailing loop. Adequate at current symbol counts. |
| Concurrency | `PositionTracker` mutex usage is correct. One real placement/cancel race window in `processExpiredPositions`. |
| Testing | Risk profile has 8 unit tests. **Most engine components have no direct unit tests** in this tree — coverage is via integration paths only. |

## Critical (3)

### CR-1: Per-trade random leverage between 2× and 20× makes risk non-deterministic and is not configurable

**Files:** [`signal_engine.cpp:161-168`](../../src/engine/signal_engine.cpp), [`signal_engine.cpp:931-943`](../../src/engine/signal_engine.cpp), [`signal_engine.cpp:170-176`](../../src/engine/signal_engine.cpp)

```cpp
constexpr int kMinOrderLeverage = 2;
constexpr int kMaxOrderLeverage = 20;

int randomOrderLeverage() {
    thread_local std::mt19937 rng(std::random_device{}());
    std::uniform_int_distribution<int> distribution(kMinOrderLeverage, kMaxOrderLeverage);
    return distribution(rng);
}
```

Called inside `openPosition`:

```cpp
const int requestedLeverage = randomOrderLeverage();
auto leverageResult = co_await m_orders.setLeverage(std::string(symbol), requestedLeverage);
```

Three independent problems:

**(a) Margin per position is non-deterministic.** With `riskPct = 0.01` (moderate profile), each trade risks 1% of available balance at ATR×slMultiplier distance — that part is deterministic. But the *margin used* per trade is `notional / leverage` ∈ `[notional/20, notional/2]`. Two positions opened seconds apart on the same signal can consume 10× different margin, exhausting `availableBalance` differently and producing different `order_cap` outcomes for subsequent trades in the same cycle.

**(b) Take-profit distance depends on leverage.** [`signal_engine.cpp:170-176`](../../src/engine/signal_engine.cpp):

```cpp
if (cfg.takeProfitPercent > 0.0) {
    const double effectiveLeverage = leverage > 0 ? static_cast<double>(leverage) : 1.0;
    return entryPrice * cfg.takeProfitPercent / (100.0 * effectiveLeverage);
}
```

A strategy configured for "take profit at 5% ROI on margin" will hit TP at +0.25% price move on 20× but +2.5% price move on 2× — a 10× spread on the same signal. The strategy's expected hold time, win/loss profile, and reward-to-risk ratio are all redrawn for each trade.

**(c) Backtesting and shadow metrics are unreproducible.** The `ShadowMetricsRecorder` records `current_price`, `atr`, `confidence` per signal but not the leverage chosen. Replay or backtest against historical signals cannot reproduce the actual outcome because the leverage was a coin-flip at runtime.

There is no `random_leverage_min` / `random_leverage_max` in `config.json`, no override flag, no log line emitting "leverage chosen X for symbol Y reason=randomization" before the order is placed. The `Logger::log` at line 945 emits the leverage *after* it was set on the exchange, in an `INFO` line.

**Recommended fix:** make leverage deterministic, derived from one of:
- The strategy config (`StrategyConfig::leverage`).
- The risk profile (`RiskProfile::leverage`).
- A function of the signal (e.g. capped by `confidence`).

If randomization is genuinely intended (e.g. to defeat exchange anti-bot heuristics), gate it behind an explicit `engine.random_leverage_enabled` flag and document why in the design doc.

### CR-2: `WorkQueue::build` evaluates only one strategy per symbol per cycle — most (strategy, symbol) pairs are silently dropped every cycle

**Files:** [`work_queue.cpp:10-52`](../../src/engine/work_queue.cpp), [`signal_engine.cpp:441-503`](../../src/engine/signal_engine.cpp)

```cpp
for (size_t symbolIndex = 0; symbolIndex < shuffledSymbols.size(); ++symbolIndex) {
    const auto* strategy = strategies[symbolIndex % strategies.size()];
    ...
    for (const auto& interval : cfg.intervals) {
        out.push_back(WorkItem{...});
    }
}
```

With **3 strategies and 100 symbols**, each cycle generates ~100 work items, not 300. Every cycle, ~67 of the (strategy, symbol) combinations are not evaluated at all.

The shuffle (lines 22, 35) means *which* pairs get skipped rotates over time, but a single cycle's signals are a random ⅓ of the actual coverage space. If a strategy's edge is "react to 5m breakouts on 100 symbols within the same scan cycle", the engine evaluates that strategy on only ~33 symbols per cycle — meaning the breakout might be missed entirely until 1–2 cycles later when the shuffle lands favorably.

The inline warning log only fires when `strategies.size() > symbols.size()` ([`work_queue.cpp:24-32`](../../src/engine/work_queue.cpp)):

```cpp
if (strategies.size() > symbols.size()) {
    const size_t unscheduled = strategies.size() - symbols.size();
    Logger::instance().log(
        LogLevel::Warning,
        "work queue per-cycle strategy rotation active unscheduled_this_cycle=" + ...
        " (probabilistically fair over time via shuffle)");
}
```

This is misleading: the warning suggests rotation only matters when strategies > symbols, but the reverse case (strategies < symbols) still silently skips most pairs.

There is no unit test for `WorkQueue::build` in this tree.

**Recommended fix:** the inner loop should iterate over `strategies` cross `symbols`, not assign one strategy per symbol:

```cpp
for (const auto* strategy : strategies) {
    if (!strategy) continue;
    const auto& cfg = strategy->config();
    for (const auto& symbol : shuffledSymbols) {
        for (const auto& interval : cfg.intervals) {
            out.push_back(WorkItem{symbol, interval, strategy});
        }
    }
}
```

If the goal of the current design is to *bound the work per cycle* (e.g. for budget/latency), make that explicit via a `maxItemsPerCycle` config and document the trade-off. The current shape is a footgun.

### CR-3: `openPosition` continues with the position open when stop-loss placement fails — naked position

**Files:** [`signal_engine.cpp:1071-1099`](../../src/engine/signal_engine.cpp)

```cpp
if (m_config.placeStopLoss || initialStopPrice > 0.0) {
    ...
    auto protectionResult = co_await m_orders.protection(ProtectionOrderDraft{...});
    if (protectionResult) {
        slResult = *protectionResult;
    } else {
        Logger::instance().log(
            LogLevel::Warning,
            "stop-loss placement failed symbol=" + std::string(symbol) +
                " error=" + quoteString(protectionResult.error().toString()));
    }
}
```

If `protection()` returns an error (network blip, exchange API rejection, invalid stop price after rounding), the code logs `Warning` and **proceeds**. The market order placed at lines 957–975 has already filled. The take-profit limit at lines 1012–1056 may or may not have placed (its failure path closes the position — good). But SL is treated as best-effort.

The fall-back is the `monitorTimeExit` loop ([`signal_engine.cpp:1209-1237`](../../src/engine/signal_engine.cpp)), which fires every `positionCheckInterval` (60s default) and only exits on `maxHoldDuration` elapsed — not on price triggers. So a position opened at $100 with no SL on a fast-moving 1h candle can lose 20% before the time-exit fires.

There is no `slResult->state != PlacementState::Accepted` check — if `protection()` returns a result but in the `Rejected` state, `slResult` is populated and the tracker stores a non-zero `slOrderId` that doesn't exist on the exchange.

Combined with `placeStopLoss=false` being the default in [`config.json:66`](../../config.json), the production system has no exchange-side protection: SL exists only if a strategy emits `initialStopPrice > 0` from the signal.

**Recommended fix:**
- If `protection()` returns `!protectionResult` *and* `m_config.placeStopLoss` was true, close the position via `closeByMarket` and unwind (cancel the TP). Same pattern used for TP failure at lines 1023–1056.
- Add `if (protectionResult->state != PlacementState::Accepted)` check, treat as failure.
- Document `placeStopLoss=false` semantics clearly: "exchange-side SL is disabled; the engine relies on `maxHoldDuration` time-exit only."

## Warning (5)

### WR-1: `processExpiredPositions` leaves position naked if `closeByMarket` fails after both protection orders were cancelled

**Files:** [`signal_engine.cpp:1310-1378`](../../src/engine/signal_engine.cpp)

Sequence:
1. Cancel TP (or skip if no TP). If cancel fails, return (TP still active — safe).
2. Cancel SL (or skip if no SL). If cancel fails, log and return — but TP is now cancelled and SL cancel failed, so the position has *one* protection if SL didn't actually cancel. Acceptable.
3. `closeByMarket(...)`. If this fails after step 1+2 succeeded, log "time-exit close failed" — but both protections are gone and the position remains open.

The post-close check at line 1369 — `if (closeSucceeded || (liveQty.has_value() && isFlatPositionQty(*liveQty)))` — covers the "already flat" race correctly. But the failure branch doesn't attempt to *re-place* either protection order, so the position is left fully naked until the next monitor cycle (60s+) when `reconcileTrackedPositions` might catch up.

**Recommended fix:** on close failure with both protections cancelled, immediately re-place at least the SL using the original `initialStopLevel` from the tracked position. Or: cancel TP and SL *after* `closeByMarket` succeeds, not before.

### WR-2: `evaluateGeminiNonBlocking` spawns a thread per call and polls every 25ms

**Files:** [`signal_engine.cpp:260-278`](../../src/engine/signal_engine.cpp)

```cpp
auto evalFuture = std::async(
    std::launch::async,
    [this, symbol = std::move(symbol), direction, signalInterval = std::move(signalInterval)]() {
        return m_geminiFilter.evaluate(symbol, direction, signalInterval, m_scanner.cache());
    });

boost::asio::steady_timer waitTimer(m_scanner.ioContext());
while (evalFuture.wait_for(std::chrono::milliseconds(0)) != std::future_status::ready) {
    waitTimer.expires_after(std::chrono::milliseconds(25));
    boost::system::error_code ec;
    co_await waitTimer.async_wait(boost::asio::redirect_error(boost::asio::use_awaitable, ec));
}
```

Each call:
- Spawns a fresh OS thread (`std::async(std::launch::async)` is required to be async by spec, unlike `std::async` without policy).
- Inside that thread, `GeminiFilterController::evaluate` may fork a Python subprocess and block for up to `timeoutSeconds` (default 45s).
- The coroutine polls every 25ms = 40 wakeups/sec while the subprocess runs.

With `maxEvaluationsPerScanCycle=3` (default), at most 3 threads per cycle. But across many cycles, this still creates many short-lived OS threads — and the 25ms polling burns ~1800 timer wake-ups per minute of subprocess time. Acceptable but unidiomatic.

**Recommended fix:** use `boost::asio::async_compose` + thread pool, or a producer/consumer queue with a fixed worker thread. Or use `co_await boost::asio::post(scanner.ioContext(), ...)` to offload the subprocess invocation. The 25ms is fine, but spawning a thread per call is not.

### WR-3: `BetaCalculator::ols` divides by `n-1` even when `n` is `coinReturns.size()` not the original sample — variance bias on truncated windows

**Files:** [`beta_calculator.cpp:102-128`](../../src/engine/beta_calculator.cpp), [`beta_calculator.cpp:85-93`](../../src/engine/beta_calculator.cpp)

The flow at lines 85–93 truncates `coinReturns` and `btcReturns` to a tail of length `n = min(window, min(coin, btc))`. Then `ols` computes:

```cpp
const double muX = accumulate(btcReturns ..., 0.0) / n;
const double muY = accumulate(coinReturns ..., 0.0) / n;
...
cov /= n - 1;
var /= n - 1;
```

Mixing population-mean denominators (`/n` for mean) with sample-variance denominators (`/(n-1)` for cov/var) is non-standard. For OLS slope, dividing both `cov` and `var` by the same constant cancels — so `cov/var` is unaffected. But the comment and the `kMinVariance = 1e-12` check apply to the *scaled* variance, not the underlying scatter. A tiny window could produce `var < 1e-12` and return NaN even when the underlying signal is meaningful.

Not strictly wrong (slope is invariant), but the helper is confusing and the NaN threshold semantics depend on `n`.

**Recommended fix:** drop both `/n-1` divisions and just compute `slope = sum_dxdy / sum_dxdx`. Adjust `kMinVariance` accordingly. Or be explicit about returning sample stats by computing them once consistently.

### WR-4: `TrailingStopController` recomputes the entire trailing decision every wake-up regardless of whether a candle closed

**Files:** [`signal_engine.cpp:1285-1308`](../../src/engine/signal_engine.cpp), [`trailing_stop_controller.cpp:10-84`](../../src/engine/trailing_stop_controller.cpp)

`processTrailingStops` runs at `trailingCheckInterval` (default 300s) and iterates all tracked positions. For each, it calls `TrailingStopController::evaluate`, which:
- Filters klines by `isClosed`.
- For default trailing, walks the last `trailingCandles` closed candles to find min low / max high.
- For swing trailing, runs an O(N×lookback²) swing detection across all closed klines.

Nothing checks "has a new candle closed since the last evaluation?" The same decision is recomputed every 5 minutes even if the most recent candle close was 50 minutes ago. The swing detection in particular is wasteful — it scans the entire `closedKlines` vector from the back, computing the same swing positions every call.

**Recommended fix:** record the last-evaluated candle openTime per position. Skip evaluation when no new closed candle is present. Cheap to track in `TrackedPosition::lastTrailingEvalCandleMs`.

### WR-5: `LossManager::placeDca` doubles down without verifying the original SL is still in place

**Files:** [`loss_manager.cpp:319-411`](../../src/engine/loss_manager.cpp)

DCA fires when `roi <= roiDcaThreshold` (default -0.80 = -80% ROI on margin). The DCA logic:
1. Cap & exposure checks for the DCA notional.
2. Place market order to grow the position.
3. Mark `state.dcaPending = true`, store pre-DCA snapshot.
4. Next evaluation: detect fill via `|positionAmt|` growth, increment `dcaCount`, clear `dcaPending`.

What's missing: the existing SL is still at the *pre-DCA* level. After DCA, the position has a higher entry price (averaged down), but the SL hasn't moved. The trade now needs to recover from -80% ROI plus pay the additional position's premium, with the same SL distance — which may be far closer to the new break-even than the old one.

The "BE flag is invalidated after DCA" logic at line 403 (`state.beCurrent = false`) ensures the *next* break-even update will re-amend — but only after the next BE threshold trigger, which requires `roi <= roiBeThreshold` (-50%). Between DCA fill and next BE update, the position is mismatched.

**Recommended fix:** after DCA fill confirmation, immediately re-evaluate and amend the SL to the new entry-price-relative level. Don't wait for the next BE threshold cross.

## Info / Style (4)

### IN-1: `signal_engine.cpp` is 1517 lines in a single file with `openPosition` spanning ~500 lines

**Files:** [`signal_engine.cpp:655-1150`](../../src/engine/signal_engine.cpp)

`openPosition` handles: sizing → order cap → exposure → risk gate → gemini gate → set leverage → market order → TP placement → SL placement → tracker commit → revert paths on every failure. Each phase is 20–80 lines and reads cleanly in isolation, but the combined function exceeds two screens and has 14 distinct `co_return` points.

Splitting into named helpers (`runPreflightGates`, `placeProtectionOrders`, `commitOrUnwind`) would make the failure-recovery semantics auditable. Today, verifying "every path that reaches the market-order placement also unwinds the tracker on later failure" requires line-by-line tracing.

Not a bug — but a maintainability cliff for a critical hot path.

### IN-2: `RiskProfile::loadActive` hardcodes "moderate" defaults in C++ but the same values exist in `config.json` — drift risk

**Files:** [`risk_profile.h:10-35`](../../src/engine/risk_profile.h), [`risk_profile.cpp:36-83`](../../src/engine/risk_profile.cpp), [`config.json:18-30`](../../config.json)

The C++ `RiskProfile` struct has default values (`riskPct=0.01`, `slMultiplier=1.5`, `softMaxDrawdown=0.20`, etc.) that mirror the `moderate` profile in `config.json`. The C++ defaults are used as fallback when:
- `active_profile` is missing/not a string.
- `active_profile` refers to an unknown profile.
- The named profile fails validation.

If `config.json` is edited to tighten the moderate profile (e.g. `riskPct: 0.005`) but C++ defaults are not updated, an operator who triggers a fallback gets the *old, looser* values silently. There is no warning that "moderate fallback" diverges from `config.json[risk_profiles.moderate]`.

**Recommended fix:** read fallback values from `config.json[risk_profiles.moderate]` directly, or document that the C++ defaults must be kept in sync. Add a startup test that compares the C++ struct to the config file.

### IN-3: `quoteString` is duplicated across 3+ files with the same implementation

**Files:** [`signal_engine.cpp:119-123`](../../src/engine/signal_engine.cpp), [`loss_manager.cpp:454-458`](../../src/engine/loss_manager.cpp), [`gemini_filter.cpp:50-54`](../../src/engine/gemini_filter.cpp), and at least 2 more in the orchestration module.

Each is:
```cpp
std::string quoteString(std::string_view value) {
    std::ostringstream out;
    out << std::quoted(std::string(value));
    return out.str();
}
```

Same in `RiskController` and elsewhere. Lift to a shared `logger_util.h` and stop maintaining copies.

### IN-4: `LossManager::nearEqual` uses a custom epsilon heuristic that's inconsistent with `kEpsilon=1e-9`

**Files:** [`loss_manager.cpp:19-29`](../../src/engine/loss_manager.cpp)

```cpp
constexpr double kEpsilon = 1e-9;

bool nearEqual(double lhs, double rhs, double epsilon) {
    return std::abs(lhs - rhs) <= std::max(epsilon, std::max(std::abs(lhs), std::abs(rhs)) * 1e-9);
}
```

The relative tolerance is `1e-9`, but the absolute floor is the caller-supplied epsilon. Callers pass `1e-8` ([`loss_manager.cpp:169`](../../src/engine/loss_manager.cpp)) for entry-price comparison and `qtyTolerance` (typically `stepSize * 0.5`) for quantity. The relative-tolerance term means a position at $50,000 entry tolerates `5e-5` price differences — but the absolute `1e-8` is irrelevant at that scale. For small-price coins (e.g. $0.001 entry) the relative tolerance is `1e-12`, smaller than typical float precision.

The function is correct but the call sites should explicitly pass the tolerance scale they want. Or use `std::nextafter`-based comparison.

## What looks good

- **`PositionTracker` locking is correct.** Single `std::mutex`, RAII `std::lock_guard`, all mutation paths take the lock. `reserve` / `commitReserved` pattern correctly prevents the open-position race.
- **`OrderCapController::sumOpenNotional` deduplicates by symbol via `std::max`.** ([`order_cap_controller.cpp:111-117`](../../src/engine/order_cap_controller.cpp)) — prevents double-counting a position appearing in both `account.positions` and the tracked set.
- **`ExposureController::decide` allows trades that *reduce* net beta deviation even when at the limit** ([`exposure_controller.cpp:191-193`](../../src/engine/exposure_controller.cpp)) — correct: a trade that brings net beta back toward target should be permitted even when current deviation is large.
- **`SizingPolicy::calculateSize` is small, pure, and handles min/max clamping cleanly.** No state, no side effects. Easy to unit-test (but isn't tested).
- **`TrailingStopController::isFavorableMove` enforces directionality correctly** — never moves a long's stop down or a short's stop up.
- **`RiskProfile::validate()` is comprehensive** — checks all cross-field invariants (`hardMinUpi <= softMinUpi`, `softMaxDrawdown <= hardMaxDrawdown`, etc.) and throws on each violation with a clear message.
- **Failure modes for `OrderCap` and `Exposure` exceptions are explicit.** Both have `OrderCapFailureMode::Closed`/`Open` toggles, and `openPosition` honors them — fail-closed by default, fail-open only if explicitly configured.
- **The Gemini cycle-gate state machine** correctly closes the gate for the rest of the cycle on quota/budget exhaustion when configured ([`signal_engine.cpp:820-844`](../../src/engine/signal_engine.cpp)), preventing further evaluations from racing with a known-failing API.
- **`LossManager::validateConfig` rejects invalid combinations** at construction and disables the manager rather than crashing. Good fail-safe.

## Suggestions for next iteration

1. **Add direct unit tests for every public engine class.** Only `RiskProfile` has tests (8 of them). `SizingPolicy`, `ExposureController`, `OrderCapController`, `TrailingStopController`, `WorkQueue`, `PositionTracker`, `LossManager` all have non-trivial logic and no tests. Even small property tests (e.g. `calculateSize` round-trip with step-size) would catch a lot.
2. **Persist the leverage chosen per trade** in the tracker (`TrackedPosition::activeLeverage`) and emit it to the shadow recorder. Without this, post-hoc analysis of why a position behaved the way it did is impossible.
3. **Move the cycle-level state out of `SignalEngine` members** (`m_geminiEvaluationsThisCycle`, `m_geminiCycleGate`, `m_lastOpenDecision`). They're modified across `runScanCycle` → `processItem` → `openPosition` and accessed via `m_lastOpenDecision` from the post-`openPosition` shadow-record block. A `CycleContext` struct passed by reference would make the dataflow explicit and prevent the next coroutine refactor from corrupting state.
4. **Document the WorkQueue rotation policy** in a design doc, with a worked example showing the expected per-strategy coverage rate at typical symbol/strategy counts.

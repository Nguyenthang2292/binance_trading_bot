## CRITICAL (immediate financial risk — fix before next live session)

### C1 — Time-exit after TP fill opens reverse position

**File:** [`signal_engine.cpp:854-877`](../../src/engine/signal_engine.cpp)

`processExpiredPositions()` logic:

```
1. canceled = cancelNormalByOrderId(tp_order_id)   ← (void) — result silently discarded
2. closeByMarket(symbol)                            ← always executed
3. m_tracker.remove(symbol)
```

When a TP limit order fills *before* the time-exit timer fires:

- Step 1: Binance returns error (order already FILLED) → discarded with `(void)`
- Step 2: `closeByMarket` is issued on a **flat position**
- In one-way mode: Binance opens a **new position in the opposite direction** with no TP/SL and no tracker entry
- The ghost position bleeds money indefinitely

**Fix:**

```cpp
// Check cancel result before closing:
auto cancelResult = co_await m_orders.cancelNormalByOrderId(pos.symbol, pos.tpOrderId);
if (!cancelResult) {
    // -2011 = unknown order (already filled/cancelled) → position already closed
    // Check error code; if already-filled, skip closeByMarket
    if (isAlreadyFilledError(cancelResult.error())) {
        m_tracker.remove(pos.symbol);
        continue;
    }
}
(void)co_await m_orders.closeByMarket(...);
```

### C2 — TP rejection not detected — position tracked without profit-taking order

**File:** [`signal_engine.cpp:700-768`](../../src/engine/signal_engine.cpp)

```cpp
auto tpResult = co_await m_orders.limit(LimitOrderDraft{...});
// ← NO check on tpResult here
// ... continues to place SL and add to tracker ...
if (tpResult && tpResult->orderId.has_value()) {  // only populates orderId
    tracked.tpOrderId = *tpResult->orderId;
}
m_tracker.add(std::move(tracked));
```

If Binance rejects the TP limit (e.g., TP price already reached in fast momentum market), the position is tracked with only an SL. No automated profit exit exists. The position runs until `maxHoldDuration` expires (or forever for snapshot-loaded positions).

**Fix:**

```cpp
auto tpResult = co_await m_orders.limit(LimitOrderDraft{...});
if (!tpResult || tpResult->state != PlacementState::Accepted) {
    Logger::instance().log(LogLevel::Error, "TP placement failed — aborting position");
    // Place immediate market close to exit the market order
    (void)co_await m_orders.closeByMarket(...);
    co_return Result<void>{};
}
```

## HIGH (significant financial or operational risk)

### H1 — Trailing stop: cancel succeeds, new SL fails → no stop protection

**File:** [`signal_engine.cpp:888-947`](../../src/engine/signal_engine.cpp)

Sequence inside `processTrailingStops()`:

```
1. cancelAlgoByAlgoId(old_sl)     → succeeds
2. m_tracker.updateStopLoss(0, {}, 0.0)  → clears SL record
3. m_orders.protection(new_sl)    → FAILS (transient API error)
4. continue  ← position now has ZERO stop protection
```

Between step 2 and step 3, if the new placement fails, the position has no downside protection and no tracker entry for any stop.

**Fix:** Don't clear the old SL from tracker until the new SL placement succeeds:

```cpp
// Keep old ID in local variable
const auto oldSlId = pos.slOrderId;
const auto oldSlClientId = pos.slClientOrderId;
// ... cancel ...
auto protection = co_await m_orders.protection(newDraft);
if (!protection || protection->state != PlacementState::Accepted) {
    // Reinstate old SL record — cancel may have failed too, but at least tracker is accurate
    m_tracker.updateStopLoss(pos.symbol, oldSlId, oldSlClientId, pos.currentTrailLevel);
    continue;
}
m_tracker.updateStopLoss(pos.symbol, protection->orderId.value_or(0), clientOrderId, decision->newLevel);
```

### H2 — TOCTOU between `has()` check and `m_tracker.add()` — duplicate market orders

**File:** [`signal_engine.cpp:635-768`](../../src/engine/signal_engine.cpp)

```
has(symbol) == false  ← second guard check at line 635
co_await m_orders.market(...)   ← coroutine yields
co_await m_orders.limit(...)    ← coroutine yields again
co_await m_orders.protection(...)
m_tracker.add(tracked)          ← only NOW marked as tracked
```

A concurrent scan cycle item for the same symbol can pass `has(symbol) == false` while the first item's market order is in-flight. Both items place market orders for the same symbol.

**Fix:** Add a sentinel/placeholder to the tracker *before* placing the market order:

```cpp
// Before m_orders.market():
TrackedPosition placeholder;
placeholder.symbol = std::string(symbol);
placeholder.direction = direction;
// Use a try-add that returns false if already present:
if (!m_tracker.tryAdd(std::move(placeholder))) {
    co_return Result<void>{};   // concurrent open already in progress
}
// ... place orders ...
// Update the placeholder with full order details on success
m_tracker.update(tracked);
// Or remove the placeholder and re-add if failed
```

### H3 — `onUserDataEvent` ignores `PARTIALLY_FILLED` — tracker quantity becomes stale

**File:** [`signal_engine.cpp:965`](../../src/engine/signal_engine.cpp)

```cpp
if (order->orderStatus != "FILLED") { return; }
```

When a TP limit order partially fills (low liquidity, large order), the position is partially closed on the exchange but the tracker retains the original full quantity. Consequences:

- Time-exit tries to close full original quantity → overshoot → opens reverse position
- Trailing stop manages wrong quantity → SL order for more contracts than are open
- Remaining TP order quantity mismatches actual position

**Fix:** Handle `"PARTIALLY_FILLED"`: subtract `executedQty` from `pos.quantity` in tracker.

### H4 — Beta calculator: kline alignment by array INDEX, not timestamp

**File:** [`beta_calculator.cpp:35-50`](../../src/engine/beta_calculator.cpp)

```cpp
const size_t klineLimit = std::min(coinKlines->size(), btcKlines->size());
for (size_t i = 1; i < klineLimit; ++i) {
    coinReturns.push_back(((*coinKlines)[i] - (*coinKlines)[i-1]) / (*coinKlines)[i-1]);
    btcReturns.push_back(((*btcKlines)[i] - (*btcKlines)[i-1]) / (*btcKlines)[i-1]);
}
```

If BTCUSDT has 365 days of data and SOLUSDT has 60 days in cache (added later or missed data during downtime), array index 0 for BTCUSDT = 305 days ago while index 0 for SOLUSDT = 60 days ago. Returns are paired by index, not by `openTime`. Beta is computed from **temporally misaligned data** — essentially noise.

All exposure controller decisions (ScaleDown thresholds, gross beta limits) are based on this fabricated beta.

**Fix:** Build a `map<openTime, return>` for each series, then compute OLS only over timestamps present in both maps.

### H5 — `runSubprocess` blocks the `io_context` thread for up to 45 seconds

**File:** `gemini_filter.cpp` (subprocess busy-poll loop)

`GeminiFilterController::evaluate()` is called synchronously (no `co_await`) from `openPosition()`. It busy-polls (Windows: `WaitForSingleObject` with 50ms timeout; POSIX: `sleep_for(20ms)`) until the Python subprocess completes — up to `timeoutSeconds = 45`.

During those 45 seconds:

- `monitorTimeExit` timer cannot fire
- `monitorTrailingStops` timer cannot fire
- No other scan cycle items are processed
- Time-sensitive position exits are delayed

**Fix:** Offload subprocess execution to `boost::asio::thread_pool`, post the result back to the `io_context`, and `co_await` the result in `openPosition()`.

### H6 — `add()` silently overwrites existing position — orphaned TP/SL orders

**File:** [`position_tracker.cpp:27-28`](../../src/engine/position_tracker.cpp)

```cpp
void PositionTracker::add(TrackedPosition pos) {
    std::lock_guard lock(m_mutex);
    m_positions[pos.symbol] = std::move(pos);  // silent overwrite
}
```

If a position for the same symbol already exists (race, double-signal, or rapid restart), the old `tpClientOrderId` and `slClientOrderId` are lost. Those orders remain live on Binance with no tracking → no automated management, no cancel on exit.

**Fix:** Return `bool` indicating collision; caller should handle accordingly:

```cpp
bool PositionTracker::add(TrackedPosition pos) {
    std::lock_guard lock(m_mutex);
    if (m_positions.contains(pos.symbol)) {
        return false;  // caller should log and abort
    }
    m_positions[pos.symbol] = std::move(pos);
    return true;
}
```

### H7 — Snapshot-loaded positions: `quantity=0`, `maxHoldDuration=0`

**File:** [`position_tracker.cpp:12-22`](../../src/engine/position_tracker.cpp)

`loadFromSnapshot()` at startup:

- Never sets `tracked.quantity` → defaults to `0.0`
- Sets `maxHoldDuration = std::chrono::seconds{0}`

Consequences:

- `expired()` checks `maxHoldDuration.count() <= 0` → skips all snapshot positions → **never time-exited**
- `processExpiredPositions`: `toDecimal(0.0)` → close-by-market never sent anyway
- No `tpClientOrderId`/`slClientOrderId` set → `onUserDataEvent` never matches fills → **tracker entry permanent**
- Snapshot positions **block new signals for those symbols indefinitely**

**Fix:** Populate `quantity` from `position.positionAmt`, and set a configurable fallback `maxHoldDuration` for snapshot-loaded positions.

## Concurrency Model Summary

| Component | Thread Safety | Notes |
|-----------|--------------|-------|
| `PositionTracker` | Safe (mutex per operation) | Correct for current usage; but `onUserDataEvent` on WS thread is safe only because it exclusively accesses mutex-protected methods |
| `ExposureController::getBeta()` | Safe (double-checked locking with re-check) | Only called from io_context thread; mutex is redundant but harmless |
| `GeminiFilterController` cache | Safe (mutex) | Called synchronously from io_context thread only |
| `m_geminiEvaluationsThisCycle` | Safe (strand-local) | Only modified/read on io_context coroutines |
| `m_running` | Safe (std::atomic) | Correct acquire/release ordering |
| co_spawn completion handlers | Safe | Run on same io_context executor |
| `runSubprocess` | **UNSAFE** (blocks io_context) | Must be offloaded — see H5 |

- `account_snapshot.h` — Data types: `AccountSnapshot`, `MarginCheckDraft`, `LiquidationRiskView`, error enums

- `rest_account_client_adapter.h/.cpp` — Adapter from `RestClient` to `IAccountRestClient`
- `mql4_account_adapter.h/.cpp` — Maps Binance Futures data to MQL4 property model

## HIGH (must fix)

### H1 — Synthesized move constructor leaves `m_rest` dangling

**File:** [`account_service.h:24-26`](../../src/account/account_service.h), [`account_service.cpp:15-18`](../../src/account/account_service.cpp)

`AccountService` has:

```cpp
std::unique_ptr<IAccountRestClient> m_ownedRest;  // line 24
IAccountRestClient& m_rest;                        // line 25 — reference member
```

The `RestClient&` constructor sets `m_ownedRest = make_unique<...>()` then binds `m_rest = *m_ownedRest`. This is correct at construction. However:

- `std::unique_ptr` causes copy-constructor to be implicitly deleted
- The **move constructor is implicitly generated**: it moves `m_ownedRest` (leaving it null) but `m_rest` still references the source's (now-null) `m_ownedRest`
- Any use of the moved-to `AccountService` dereferences a null pointer → **undefined behavior / crash**

No `= delete` on copy/move anywhere in the class.

**Fix:**

```cpp
// account_service.h — add in class body:
AccountService(const AccountService&) = delete;
AccountService& operator=(const AccountService&) = delete;
AccountService(AccountService&&) = delete;
AccountService& operator=(AccountService&&) = delete;
```

**Also:** Member declaration order (`m_ownedRest` before `m_rest`) is critical for the constructor to work. Add a comment:

```cpp
std::unique_ptr<IAccountRestClient> m_ownedRest;  // must be declared before m_rest
IAccountRestClient& m_rest;
```

### H2 — `accountLeverage()` fails when no position is open

**File:** [`mql4_account_adapter.cpp:162-179`](../../src/account/mql4_account_adapter.cpp)

```cpp
AccountMappingResult<int64_t> Mql4AccountAdapter::accountLeverage(std::string symbol) const {
    // searches snapshot.positions for the symbol...
    return std::unexpected(AccountMappingError::SnapshotIncomplete);
}
```

This only succeeds when a **position is currently open** for the symbol. EAs typically call `AccountLeverage()` at startup to compute lot sizes — before opening any position. This always returns `SnapshotIncomplete`, breaking all leverage-based position sizing at startup.

Binance `/fapi/v2/positionRisk` returns leverage even when `positionAmt == 0`. These zero-position entries need to be included in the snapshot.

**Fix:** Ensure `positionRisk` entries with `positionAmt == 0` are included in the snapshot's `positions` vector (currently they may be filtered out upstream). Alternatively, add a dedicated `GET /fapi/v2/leverageBracket` query path.

## MEDIUM (should fix)

### M1 — `co_return std::unexpected(...)` implicit variant conversion

**File:** [`account_service.cpp:22,27,42,56,75`](../../src/account/account_service.cpp)

```cpp
// Function returns AccountServiceResult<T> where error = AccountServiceError
// = std::variant<BinanceError, AccountMappingError>
co_return std::unexpected(AccountMappingError::Unsupported);  // AccountMappingError, not variant
```

This compiles under C++23 `std::expected` (which has a converting constructor from `unexpected<G>` when `E` is constructible from `G`). However, the custom fallback `expected_compat.h` in this codebase lacks those converting constructors — **it will fail to compile if the fallback is active**.

**Fix:** Be explicit on all five sites:

```cpp
co_return std::unexpected(AccountServiceError{AccountMappingError::Unsupported});
co_return std::unexpected(AccountServiceError{acc_res.error()});
```

### M2 — `availableBalance` includes pending-order margin; MQL4 FreeMargin does not

**File:** [`mql4_account_adapter.cpp:159`](../../src/account/mql4_account_adapter.cpp)

Binance: `availableBalance = marginBalance - initialMargin - openOrderInitialMargin`
MQL4: `AccountFreeMargin() = Equity - Margin` (positions only, not pending orders)

When open orders exist, free margin is underreported. EAs checking `AccountFreeMargin()` before placing new orders may incorrectly refuse entries.

**Fix:** Compute as `marginBalance - initialMargin` using raw `Balance` struct fields.

### M3 — MarginLevel exact `== 0.0` comparison

**File:** [`mql4_account_adapter.cpp:67`](../../src/account/mql4_account_adapter.cpp)

```cpp
if (*margin == 0.0) { return 0.0; }
return (*equity / *margin) * 100.0;
```

- Only catches exact IEEE zero — subnormal/near-zero values bypass the guard and produce astronomically large or garbage MarginLevel values
- Returns `0.0` conflates "no positions open" (safe) with "data error" (dangerous)

**Fix:**

```cpp
if (std::abs(*margin) < 1e-8) {
    return std::unexpected(AccountMappingError::Unsupported);  // or dedicated NoOpenPositions error
}
```

### M6 — `accountLeverage` symbol comparison is case-sensitive

**File:** [`mql4_account_adapter.cpp:169,175`](../../src/account/mql4_account_adapter.cpp)

```cpp
if (position.symbol == symbol) { return position.leverage; }
```

All other asset comparisons in the file go through `toUpper()`. If caller passes `"btcusdt"` and position stores `"BTCUSDT"`, lookup silently fails and returns `SnapshotIncomplete`.

**Fix:**

```cpp
if (toUpper(position.symbol) == toUpper(symbol)) { return position.leverage; }
```

## What's Good

- **Coroutine usage** — `co_await`/`co_return` pattern is correct; local variables survive suspension via coroutine frame
- **Error discrimination** — `AccountServiceError = variant<BinanceError, AccountMappingError>` cleanly separates network vs. semantic errors
- **Dependency injection** — dual-constructor pattern enables testing without real REST calls
- **`Mql4AccountAdapter`** — value semantics (snapshot by value) eliminates lifetime concerns for the adapter
- **`displayAssetBalance` fallback chain** — checks `/fapi/v2/balance` first, then falls back to `/fapi/v2/account` assets
- **`accountCredit()` policy** — explicit `AssumeZero` opt-in for the zero-credit fallback is the right design

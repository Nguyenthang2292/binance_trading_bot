## Module Overview

The `src/catalog` module handles plugin discovery, integrity verification, loading, and strategy registration:

- [`plugin_handle.h`](../../src/catalog/plugin_handle.h) / [`plugin_handle.cpp`](../../src/catalog/plugin_handle.cpp) ‚Äî RAII wrapper around DLL/SO handles (`LoadLibraryExW` / `dlopen`), resolves exports
- [`plugin_loader.h`](../../src/catalog/plugin_loader.h) / [`plugin_loader.cpp`](../../src/catalog/plugin_loader.cpp) ‚Äî Enumerates plugins, SHA-256 allowlist enforcement, path confinement, creates strategy instances
- [`strategy_catalog.h`](../../src/catalog/strategy_catalog.h) / [`strategy_catalog.cpp`](../../src/catalog/strategy_catalog.cpp) ‚Äî Orchestrates loading + registry population from JSON config
- [`catalog_reporter.h`](../../src/catalog/catalog_reporter.h) / [`catalog_reporter.cpp`](../../src/catalog/catalog_reporter.cpp) ‚Äî CLI listing + startup/runtime diagnostics logging

This is a **maintenance review**: v1 closed all critical/high items. This pass re-verifies those fixes and looks for new issues, robustness gaps, and test coverage holes.

## MEDIUM (should fix)

### M1 ‚Äî Non-string `type` in strategy config throws an uncaught exception, aborting startup

**File:** [`strategy_catalog.cpp:37`](../../src/catalog/strategy_catalog.cpp)

```cpp
const std::string type = item.value("type", "");
```

`nlohmann::json::value(key, default)` throws `type_error.302` when the stored value at `key` is type-incompatible with the default. If `config.json` contains `"type": 123` (or a bool/array/object), this throws.

`initialize()` has no try/catch, and its caller ‚Äî [`main.cpp:562`](../../src/main.cpp) ‚Äî is **not** inside a try block (the only `try` there wraps config-file parsing, lines 510‚Äì517). With `std::set_terminate(terminateHandler)` active ([`main.cpp:506`](../../src/main.cpp)), the result is `std::terminate` ‚Üí crash on startup.

This is inconsistent with the surrounding code, which deliberately collects *every other* malformed-config case into `summary.errors` and continues (missing `type`, non-object entry ‚Äî [`strategy_catalog.cpp:33-41`](../../src/catalog/strategy_catalog.cpp)). A wrong-typed `type` field should be the same graceful error, not a hard abort.

**Fix:** guard the type read, e.g.

```cpp
if (!item.contains("type") || !item["type"].is_string()) {
    summary.errors.push_back("strategy config 'type' must be a string");
    continue;
}
const std::string type = item["type"].get<std::string>();
```

### M2 ‚Äî SHA-256 integrity check has a TOCTOU window before load

**Files:** [`plugin_loader.cpp:173`](../../src/catalog/plugin_loader.cpp) (verify) ‚Üí [`plugin_loader.cpp:179`](../../src/catalog/plugin_loader.cpp) (load)

`verifyIntegrity()` opens and hashes the file via `calculateSha256()` ([`plugin_loader.cpp:310-351`](../../src/catalog/plugin_loader.cpp)), then `m_loadFn(path)` re-opens it through `LoadLibraryExW`/`dlopen`. Between those two opens, an attacker with write access to the (confined, absolute) plugins directory can swap the file ‚Äî the bytes that get hashed are not guaranteed to be the bytes that get mapped.

**Scope:** the attacker must already have write access to the plugins directory, which is the same trust boundary the allowlist defends, so this is defense-in-depth rather than a primary control. Still worth documenting and, ideally, hardening.

**Fix options (in order of effort):**

- Document the limitation explicitly so operators don't over-trust the allowlist alone (also rely on directory ACLs).
- Hold the file open / take a read lock across hash+load so it cannot be replaced mid-window.
- Hash the in-memory image after mapping where the platform allows.

## LOW (maintainability / hardening)

### L1 ‚Äî `PluginHandle` metadata fields are public and mutable

**File:** [`plugin_handle.h:36-38`](../../src/catalog/plugin_handle.h)

```cpp
std::filesystem::path path;
std::string type;
std::string version;
```

v1 fixed the function pointers (H4) but left `type` public and non-const. `type` is the lookup key in `createStrategy` ([`plugin_loader.cpp:199-201`](../../src/catalog/plugin_loader.cpp)); any code holding a `PluginHandle&` can silently rewrite it and break dispatch. Lower risk than the function pointers, but for consistency these should be private with `[[nodiscard]]` const accessors, set only by `load()`/`fromExports()`.

### L2 ‚Äî `PluginLoader::loadAll()` reload invalidates outstanding strategies (undocumented contract)

**File:** [`plugin_loader.cpp:165-167`](../../src/catalog/plugin_loader.cpp)

`loadAll()` calls `m_handles.clear()`, which unloads the DLLs. Any `IStrategy` previously produced by `createStrategy` ‚Äî whose deleter and vtable live in those DLLs ‚Äî becomes dangling. `StrategyCatalog::initialize` handles this correctly by clearing the registry first ([`strategy_catalog.cpp:20`](../../src/catalog/strategy_catalog.cpp)), but a direct `PluginLoader` user has no such guardrail. This is the C1 footgun one layer down.

**Fix:** document the lifetime contract on `loadAll()` ("invalidates all instances from prior loads ‚Äî release them first"). No code change required if the catalog stays the only caller.

### L3 ‚Äî `logRuntimeStatus` parameter named `activeSymbols` but logged as `queue=`

**Files:** [`catalog_reporter.h:18-21`](../../src/catalog/catalog_reporter.h), [`catalog_reporter.cpp:64-72`](../../src/catalog/catalog_reporter.cpp)

The parameter is `activeSymbols`; the call site passes `queueItems` ([`main.cpp:1443`](../../src/main.cpp)) and it's logged as `queue=`. Three different names for one value. Pick one (`queueItems`) across the signature and the log string to avoid confusion when reading logs.

### L4 ‚Äî `isWithinDirectory` is case-sensitive on Windows

**File:** [`plugin_loader.cpp:86-95`](../../src/catalog/plugin_loader.cpp)

Component comparison uses `*rootIt != *candidateIt`, which is case-sensitive. `weakly_canonical` does not normalize case on Windows. Today both operands derive from the same `pluginsDir`, so casing is consistent and confinement works ‚Äî but if a future caller supplies a differently-cased root (e.g. from user config vs. enumeration), a legitimate plugin could be wrongly rejected. Consider a case-insensitive compare on `_WIN32`.

### L5 ‚Äî Move constructor initializer-list order does not match declaration order (`-Wreorder`)

**File:** [`plugin_handle.cpp:106-120`](../../src/catalog/plugin_handle.cpp)

The initializer list orders `m_createFn‚Ä¶m_versionFn` before `path/type/version`, but declaration order (header) is `path/type/version` first, then the private members. Members initialize in declaration order regardless, so this is **not** a functional bug (all initializers read independent `other.*` fields). It does emit `-Wreorder`. The root [`CMakeLists.txt`](../../CMakeLists.txt) does not enable `-Wall`/`-Werror`, so the build is unaffected today ‚Äî but reorder it to match declaration order to keep the file clean if warnings are tightened later.

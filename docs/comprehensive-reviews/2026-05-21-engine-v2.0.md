## Overview

This is a **follow-up** review. The first pass raised 12 findings; **all 3 critical and all 5 warning findings are confirmed resolved**, and 1 of 4 info items is addressed. The most impactful single change is that production behavior now matches what `config.json` and `RiskProfile` would lead a reader to expect: leverage is deterministic by default (from `StrategyConfig::leverage{10}`), the work queue evaluates every (strategy, symbol) pair per cycle, and stop-loss placement failure correctly unwinds the position instead of leaving it naked.

A second major delta is testing: every previously-untested engine class now has direct unit-test coverage. `test_signal_engine.cpp` ballooned from a handful of cases to 80 KB of scenarios; `test_loss_manager.cpp` is 15 KB and covers DCA fill detection, BE amend, DCA-then-SL refresh; `test_work_queue.cpp` is 12 KB. New files cover `BetaCalculator`, `ExposureController`, `OrderCapController`, `PositionTracker`, `SizingPolicy`, `TrailingStopController` — closing the entire coverage gap flagged in v1.0.

The signal engine grew from 1517 to 1643 lines — expected, since most fixes added unwind logic — but the file is still the single biggest source in the module and `openPosition` remains long. That structural refactor (v1.0 IN-1) was not attempted and is left as a follow-up.

## New Findings (1)

### NW-1: `requestedOrderLeverage` does not validate `randomLeverageMin <= randomLeverageMax` at config-construction time

**Files:** [`signal_engine.cpp:173-194`](../../src/engine/signal_engine.cpp), [`signal_engine.h:79-81`](../../src/engine/signal_engine.h), [`src/main.cpp`](../../src/main.cpp)

```cpp
std::pair<int, int> randomOrderLeverageRange(const SignalEngine::Config& engineConfig) {
    int minLeverage = clampOrderLeverage(engineConfig.randomLeverageMin);
    int maxLeverage = clampOrderLeverage(engineConfig.randomLeverageMax);
    if (minLeverage > maxLeverage) {
        std::swap(minLeverage, maxLeverage);
    }
    return {minLeverage, maxLeverage};
}
```

If an operator misconfigures `random_leverage_min: 20, random_leverage_max: 2` in `config.json`, the helper silently swaps them. That's "lenient" but means no error is surfaced — the operator never finds out the config is wrong. Also, if both are equal to `0` or set to negative, `clampOrderLeverage(0) = 1`, so the user gets `[1, 1]` and every trade is 1×, which silently defeats the random-leverage feature.

**Suggested fix:** validate in `main.cpp` when constructing `engineConfig`. Reject `randomLeverageEnabled=true` with `randomLeverageMin > randomLeverageMax` (after clamp) as a startup error, or at minimum log a WARN line at startup with the resolved range. The current behavior is safe — but it's a footgun for the next operator who edits config.json.

Info-level. Not blocking.

## What looks good (now)

- **Leverage is now reproducible.** `StrategyConfig::leverage{10}` is the deterministic default; randomization is opt-in via `engineConfig.randomLeverageEnabled`. The active leverage is persisted into `TrackedPosition::activeLeverage` so post-mortem and backtest replays can use the same value. Win.
- **`WorkQueue::build` is the obvious cross product.** Reads in ten lines. The shuffle still provides per-cycle ordering randomness without changing the *set* of items evaluated.
- **`processExpiredPositions` close-first ordering is the correct safety pattern.** If `closeByMarket` fails, both protection orders remain on the exchange and the next reconcile cycle will catch up. Position is never naked.
- **`refreshStopLossAfterDca` preserves the geometric meaning of the stop.** "Same distance from the (new) entry" is the right semantic for an averaged-down position.
- **Gemini eval pool replaces ad-hoc threads.** `m_geminiEvaluationPool` is owned by the engine, sized by config, and the completion handler resumes on the right executor. Standard boost::asio idiom.
- **Trailing-stop skip is one comparison.** `if (*latestClosed <= pos.lastTrailingEvalCandleMs) continue;` — no recomputation when nothing has happened.
- **The test suite now covers ~every public engine surface.** This is the difference between "trust the code" and "verify the code". The latter is now possible.

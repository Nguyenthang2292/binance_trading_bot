## HIGH (must fix before wiring)

### H1 — `AccountSnapshotCompleteness::Full` overstates when only `includeAccountConfig` is true

**File:** [`account_service.cpp:86-95`](../../src/account/account_service.cpp)

```cpp
if (request.includeAccountConfig) {
    auto config_res = co_await m_rest.accountConfig();
    if (!config_res) {
        co_return std::unexpected(AccountServiceError{config_res.error()});
    }
    snapshot.account.canTrade = config_res->canTrade;
    snapshot.dualSidePosition = config_res->dualSidePosition;
    snapshot.multiAssetsMargin = config_res->multiAssetsMargin;
    snapshot.completeness = AccountSnapshotCompleteness::Full;   // ← unconditional
}
```

The branch unconditionally sets `Full` regardless of whether balance/positions were also fetched. A caller that sets `includeAccountConfig=true` but leaves `includeBalanceEndpoint=false` and `includePositions=false` receives a snapshot marked `Full` while `snapshot.balances` and `snapshot.positions` are both `std::nullopt`.

**Impact today:** No engine caller sets `includeAccountConfig=true` (verified via grep), so the misclassification is dormant.

**Impact tomorrow:** The semantic ladder `AccountOnly → AccountAndBalance → AccountAndPositions → AccountBalanceAndPositions → Full` is the contract that downstream callers will use to decide whether to dereference `snapshot.balances`/`snapshot.positions`. `Full` must imply ALL data is present. Today's code violates that contract.

**Fix (preferred):** Only advance to `Full` when all three flags were honored:

```cpp
if (request.includeAccountConfig) {
    auto config_res = co_await m_rest.accountConfig();
    if (!config_res) {
        co_return std::unexpected(AccountServiceError{config_res.error()});
    }
    snapshot.account.canTrade = config_res->canTrade;
    snapshot.dualSidePosition = config_res->dualSidePosition;
    snapshot.multiAssetsMargin = config_res->multiAssetsMargin;
    if (snapshot.completeness == AccountSnapshotCompleteness::AccountBalanceAndPositions) {
        snapshot.completeness = AccountSnapshotCompleteness::Full;
    }
    // else: keep the lesser state — config alone does not promote to Full
}
```

**Alternative:** Introduce a separate boolean `snapshot.hasAccountConfig` (or new enum value `AccountWithConfig`) so completeness and config presence are orthogonal.

**Test gap:** [`tests/test_account_service.cpp:79`](../../tests/test_account_service.cpp) (`SnapshotIncludeAccountConfigLoadsModeFlags`) exercises the case but does **not** assert the documented invariant that `Full` implies balances+positions. Add a regression test for the `includeAccountConfig`-only path.

### M1 — `checkFreeMargin` silently downgrades `ServerValidatedOnly` to `Estimated`

**File:** [`account_service.cpp:128-154`](../../src/account/account_service.cpp)

When both `draft.useServerTestOrder` and `draft.assumedPrice` are provided:

```cpp
if (draft.useServerTestOrder) {
    ...
    result.completeness = MarginCheckCompleteness::ServerValidatedOnly;  // strong
    result.serverAccepted = true;
}

if (draft.assumedPrice) {
    ...
    if (leverage > 0) {
        ...
        result.completeness = MarginCheckCompleteness::Estimated;        // weaker, overwrites
    }
}
```

`Estimated` overwrites `ServerValidatedOnly`. Callers checking the enum can't distinguish "server said yes AND we have an estimate" from "we only have an estimate". The serverAccepted flag survives, but it's not exposed in the completeness type system.

**Fix:** Either keep the strongest assertion (don't overwrite if already `ServerValidatedOnly`), or split into two independent fields:

```cpp
// Option A — precedence
if (result.completeness == MarginCheckCompleteness::Unavailable) {
    result.completeness = MarginCheckCompleteness::Estimated;
}
// (keep ServerValidatedOnly if already set)
```

```cpp
// Option B — orthogonal flags in MarginCheckResult
bool serverValidated{false};
bool estimateAvailable{false};
```

**Test gap:** [`test_account_service.cpp:195`](../../tests/test_account_service.cpp) covers test-order-only; no test exists for "both server + assumedPrice".

### M2 — `accountMargin()` and `accountFreeMargin()` use different sources → MQL4 invariant breaks

**File:** [`mql4_account_adapter.cpp:190-203, 227-242`](../../src/account/mql4_account_adapter.cpp)

```cpp
AccountMappingResult<double> Mql4AccountAdapter::accountFreeMargin() const {
    ...
    return (*balance)->marginBalance - (*balance)->initialMargin;   // asset-level
}

AccountMappingResult<double> Mql4AccountAdapter::accountMargin() const {
    ...
    const auto& positions = positionsForMapping(m_snapshot);
    if (!positions.empty()) {
        return positionsInitialMargin(positions);                    // positions-summed
    }
    return (*balance)->initialMargin;                                // asset-level fallback
}
```

In MQL4 the invariant `AccountFreeMargin() == AccountEquity() - AccountMargin()` is sacred. Our mapping breaks it whenever the position-summed initial margin differs from the asset-level `initialMargin`:

- After v1 fix H2, snapshots can include zero-positionAmt entries (leverage-only). Their `initialMargin` contributes 0, so the sum may equal the asset value — but only if Binance and our parser agree exactly.
- Pending order margin is in `balance.initialMargin` (per Binance docs) but NOT in `position.initialMargin`. As soon as the user has open orders, the two diverge.
- `accountInfoDouble(MarginLevel)` (line 98-110) computes `equity/margin` using `accountMargin()` — so MarginLevel also drifts when the two sources disagree.

**Reproducer scenario:** snapshot has 1 open BTCUSDT position with `initialMargin=100` AND 1 pending limit order reserving `50` initial margin → `balance.initialMargin=150`, `positionsInitialMargin=100`. The adapter reports:

- `accountMargin()` = 100
- `accountFreeMargin()` = `marginBalance - 150`
- `accountEquity() - accountMargin()` = `marginBalance - 100` ≠ FreeMargin

**Fix:** Pick one source and use it consistently. Asset-level `initialMargin` is recommended because:

1. It includes pending-order reserve (matches MQL4 semantics: positions, not orders — but if you compute Equity-Margin you implicitly require both to come from the same denominator).
2. The v1 fix M2 deliberately switched FreeMargin to use it.

```cpp
AccountMappingResult<double> Mql4AccountAdapter::accountMargin() const {
    if (auto mode = requireSingleAssetMode(m_snapshot); !mode) {
        return std::unexpected(mode.error());
    }
    const auto balance = displayAssetBalance(m_snapshot);
    if (!balance) {
        return std::unexpected(balance.error());
    }
    return (*balance)->initialMargin;   // unified source
}
```

If summed-position margin is intentionally desired (e.g., to exclude pending-order margin from "Margin"), document the deviation from MQL4 in code and tests.

**Test gap:** No test exercises a snapshot where `balance.initialMargin != Σpositions.initialMargin`. Add one.

### M3 — `checkFreeMargin` returns success/`Unavailable` when both options are false

**File:** [`account_service.cpp:100-157`](../../src/account/account_service.cpp)

```cpp
MarginCheckDraft draft;  // useServerTestOrder=true (default), assumedPrice=nullopt
// caller flips useServerTestOrder=false and forgets to set assumedPrice
auto r = co_await svc.checkFreeMargin(draft);
// r is success, r->completeness == Unavailable, no error
```

The function silently returns `MarginCheckResult{completeness=Unavailable}` instead of signalling that no check could be performed. Callers using `if (r) use(*r);` get a useless value.

**Fix:** Reject the empty-option case at the entry:

```cpp
if (!draft.useServerTestOrder && !draft.assumedPrice) {
    co_return std::unexpected(AccountServiceError{AccountMappingError::Unsupported});
}
```

This mirrors v1 M4's spirit (consistent stub contracts) — `checkFreeMargin` should not succeed when it does nothing.

### M4 — `snapshot.account.canTrade` silently overwritten by `accountConfig.canTrade`

**File:** [`account_service.cpp:91`](../../src/account/account_service.cpp)

```cpp
// account_service.cpp:50  ─ from /fapi/v2/account
snapshot.account = std::move(*acc_res);
...
// account_service.cpp:91  ─ from /fapi/v1/accountConfig (only if requested)
snapshot.account.canTrade = config_res->canTrade;
```

Two endpoints return the same `canTrade` field. The second silently wins when `includeAccountConfig=true`. If the two endpoints disagree (stale data, partial outage, account-level vs config-level distinction), the user sees only the config value with no warning.

**Risk:** Minor today (engine doesn't request `includeAccountConfig`), but the override should be deliberate. Binance treats them as independent flags (account-level "can trade" vs config-level "can trade"). Merging them loses one bit of information.

**Fix:** Either:

- AND them: `snapshot.account.canTrade = snapshot.account.canTrade && config_res->canTrade;` (most conservative)
- Add a separate `snapshot.configCanTrade` and keep them distinct.

**Test gap:** [`test_account_service.cpp:82`](../../tests/test_account_service.cpp) tests the overwrite (sets `canTrade=false` in config and asserts the snapshot value) but doesn't cover the disagreement case.

- **Helper namespaces** — `account::mql4::` sub-namespace cleanly separates the MQL4 compatibility layer from core service code.
- **Test coverage on adapter** — 13 test cases in `test_mql4_account.cpp` exercise core mappings, multi-asset mode, missing display asset, zero-leverage edge cases, AssumeZero credit policy.
- **`displayAssetBalance` fallback chain** — Still correct: checks `/fapi/v2/balance` array first, then `/fapi/v2/account` assets. The new `requireSingleAssetMode` guard runs before either.

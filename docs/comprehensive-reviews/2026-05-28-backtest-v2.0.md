### 1. 🟡 Contradictory window-candle defaults (`backtest_gate.h:78-79`)

```cpp
int windowMinCandles{2000};   // floor on required bars
int windowMaxCandles{1500};   // cap on required bars
```

`requiredBars` is computed as `min(max(windowMinCandles, mult*slowest), windowMaxCandles)`
(`backtest_gate_controller.cpp:328-333`), and the startup buffer check repeats the same
`std::min(windowMinCandles, windowMaxCandles)` pattern (`main.cpp:1321-1323`). With the
defaults, `min(2000, 1500) = 1500`, so the "minimum" of 2000 can never be honored — you
end up requiring **fewer** bars than the declared floor, silently.

- **Production impact is limited:** `config.json:211-212` overrides to `500 / 1500`
  (consistent). The bug only bites callers that rely on defaults or set `min > max`.
- **No diagnostic:** nothing warns when `windowMinCandles > windowMaxCandles`.

**Recommendation:** Make the defaults consistent (e.g. `windowMinCandles{1500}` or raise
the max), and add a one-line startup warning in `main.cpp` when `windowMinCandles >
windowMaxCandles` so misconfiguration is visible instead of silently clamped.

### 2. 🔵 Dead code: `ParameterSpace::calculateTotalCombos`

Declared at `parameter_space.h:47-49`, defined at `parameter_space.cpp:165-175`, **zero
callers** across `src/` and `tests/`. `clampToBudget` counts via `countGridRecursiveCapped`
directly. Remove both the declaration and definition to reduce surface area.

### 3. 🔵 Stale developer comment (`plateau_finder.cpp:12-18`)

The header comment block reads like scratch notes mid-thought ("`However, the v2.0 design
says...`", "`Wait, the grid might have varying steps, but...`"). The implementation that
follows (index-based dimension matching) is correct and was hardened in v1.0; the comment
should be rewritten to describe *what the code does* in one or two sentences, not the
author's reasoning process.

### 4. 🔵 Duplicate formatters (`indicator_adapters.cpp:91-101`)

```cpp
std::string fmtPrice(double v)   { ... setprecision(2) ... }
std::string fmtPercent(double v) { ... setprecision(2) ... }  // identical body
```

Two names suggesting different formatting, one implementation. Either collapse to a single
helper or make `fmtPercent` actually differ (e.g. append `%` / different precision) so the
names stop lying. Cosmetic, but it's exactly the kind of drift that causes confusion later.

### 5. 🔵 Eval-dir accumulation over process lifetime (`gemini_range_proposer.cpp`)

`cleanupStaleEvalDirsOnce()` is guarded by `m_cleanupDone` and runs only on the **first**
`propose()` call, sweeping dirs older than `staleRuntimeTtlHours`. Each subsequent
proposal creates `runtimeDir/eval-<id>/{input,output}.json` (`:606-608`) that is never
removed during the session. For a bot running for days/weeks with the gate active, these
dirs grow unbounded until the next restart.

**Recommendation:** either remove the eval dir on a successful parse (keep-on-failure for
debugging is reasonable), or run the stale sweep periodically rather than once. Low impact
(small JSON files) but worth tidying for a long-lived process.

## ⚪ Notes (intentional / not defects, documented for maintainers)

- **`profitFactor` infinity sentinel** (`backtest_engine.cpp:179-180`): returns
  `std::numeric_limits<double>::infinity()` when there are wins but zero losses. The only
  consumer (`scoreGrid`) guards with `std::isfinite` (`backtest_gate_controller.cpp:180`),
  so it's safe — but any future consumer of `BacktestStats::profitFactor` must do the same.
- **Sharpe/Sortino are non-annualized and use population variance** — documented in the
  header and intentional, since they're only used for *relative* ranking across combos.
- **Golden-cross confidence floor** (`indicator_adapters.cpp:211`):
  `clamp(spread/0.01, baseConfig.minConfidence, 1.0)` lower-bounds confidence at
  `minConfidence`, so the engine's `sig.confidence < minConfidence` reject
  (`backtest_engine.cpp:142`) can never fire for this adapter. Appears intentional for
  plugin parity; flagged only so it isn't mistaken for a missing filter.
- **`survivors[i].at(k)` in center computation** (`plateau_finder.cpp:266`): assumes every
  survivor shares the full dimension set. Safe because all points come from
  `ParameterSpace::grid` with identical ranges; would throw `std::out_of_range` only if
  that invariant were ever broken.

  identical outputs" is honored, with parity tests backing it.
- **Prompt-context isolation enforced** — `proposeWithPartitions` reads only
  `promptContext`; calibration/OOS/signal bar are structurally inaccessible, preventing
  look-ahead leakage into the proposer.

  a `PageFetcher`, so off-by-one/budget/timeout logic is unit-tested without sockets.
- **Robust cross-platform subprocess** — proper pipe draining, timeout→terminate/kill, and
  output-file read even on non-zero exit so structured Python failures survive.
- **Clean orchestration** — `evaluate()` reads as a 10-phase pipeline with delegated
  helpers (`scoreGrid`, `runVote`, `buildPassResult`, `evaluateCached`) and structured
  JSON logging at every phase.
- **Amortized SMA** — the `thread_local` prefix-sum cache turns per-call SMA from O(period)
  to O(1) with carefully chosen reset triggers (data pointer, first-bar fingerprint,
  size sanity).
